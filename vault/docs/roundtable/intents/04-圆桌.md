# 意图：/圆桌

**触发方式：** `/圆桌` 或 `圆桌、`、`讨论、`（后接主题）

**别名：** `/讨论` `/brainstorm`

**触发时机：** 复杂问题需要专家讨论

**协议版本：** v2（2026-03-04）— 详见 `docs/roundtable/roundtable-v2-protocol.md`

---

## 作用

召集虚拟专家，输出方案对比，辅助决策。采用**上下文注入 → 发散想法 → 提取想法 → 固化规则**四阶段流程。

---

## 执行流程（v2）

```
1. 识别问题类型（效率/质量/流程/架构）
2. 【新增】输出推荐专家名单 + 理由 + 缺失风险 → 用户确认/增补
3. 未确认不进入正式讨论；确认后，先进行「业务上下文注入」（读取文档/追问痛点约束）
4. 基于拉齐的上下文，进入「发散想法」阶段。**AI必须作为发散主力，保底主动输出 3 个截然不同的解决方向，禁止在此时反问用户“您有什么想法”或依赖用户抛出观点。强制引入“红队（唱反调）”角色进行观点碰撞。**
5. 发散 → 提取：输出方案对比表 + 待确认问题（≤5 个）+ 建议决议
6. 用户明确确认后：固化规则，写入 consensus.md
```

---

## 专家推荐与确认（v2 必执行）

**AI 必须先输出推荐名单，等待用户确认：**

| 用户说 | 含义 |
|--------|------|
| 确认 | 按推荐名单进入讨论 |
| 增补：角色名 | 在名单上追加 |
| 调整：说明 | 移除或替换某位 |
| 跳过 | 使用默认推荐（仅在紧急时） |

---

## 专家路由规则（默认推荐，可被用户增补）

| 问题类型 | 召集专家 |
|----------|----------|
| 效率问题 | 流程专家 + AI 协作专家 + 执行角色 |
| 质量问题 | 边界检查员 + 审查专家 + 测试专家 |
| 流程设计 | 流程专家 + 执行角色 + 边界检查员 |
| 架构设计 | 全栈专家 + 边界检查员 + PM |

## 示例（v2 流程）

**用户：** `/圆桌 订单审单流程优化`

**AI（第一步 - 专家推荐）：**
```
[人机协作-圆桌] 收到。这是效率问题。

## 专家推荐

| 序号 | 专家角色 | 必要性 |
|------|----------|--------|
| 1 | 流程专家 | 负责流程拆解和边界识别 |
| 2 | AI 协作专家 | 负责人机分工和追问策略 |
| 3 | 审核员（执行角色） | 负责落地视角和约束条件 |

**若缺某角色可能导致：** 漏掉系统限制或执行细节。

**可增补位：** 你可追加 1–2 位（如 PM、ERP 对接人）。

请确认名单：[确认 / 增补：角色名 / 调整：说明]
```

**用户：** 确认

**AI（第二步 - 业务上下文注入）：**
```
[人机协作-圆桌] 专家已确认。在发散前，我们需要拉齐业务上下文：

**当前已知：**
- 订单来源：天猫、京东、抖店
- 核心痛点：[需要你补充，比如"审单人手不足"还是"异常单拦截不准"？]
- 客观约束：老系统（ERP）不支持实时并发写入

请确认或补充以上上下文。
```

**用户：** 核心痛点是异常单（如地址错误、超重）拦截不准，导致后端发货成本高。约束加上：不能增加人工编制。

**AI（第三步 - 发散+提取）：**
```
[人机协作-圆桌] 上下文已拉齐。进入讨论。

（发散阶段：各专家基于上下文发言...）

## 方案对比表
（标准格式，≥3 方案）

## 待确认问题
（≤5 个）

## 建议决议
（1 句话）

请确认后，我将写入 consensus.md。
```

## 注意事项

- 方案对比表必须有 3 个方案
- 待确认问题不超过 5 个
- 必须有推荐指数
- **AI 必须作为发散主力**：在发散阶段，AI 必须主动提供至少 3 个差异化的解决方向，并强制引入极端/对立（红队）视角进行内部观点碰撞。**绝对禁止**在给出充足方案前反问用户“您的想法是？”。将填空题变成选择题/改错题。
- **不主动退出**：在圆桌/头脑风暴模式下，只有在用户明确说出「结束」「进入下一阶段」或下达其他指令时，AI 才能结束头脑风暴；不得自行切换到策划、设计或实现阶段
- **必须有产出**：圆桌必须按流程要求输出方案对比表、待确认问题等规定产出
- **状态标识**：圆桌模式下每次回复开头必须带 `[人机协作-圆桌]`
