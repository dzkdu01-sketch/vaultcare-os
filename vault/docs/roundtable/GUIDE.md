# 圆桌使用指南（引导式）

## 降低输入门槛：你只需说主题

**不需要写背景/约束/期望**，我会用步进式问题引导你。

---

## 引导流程（4 步，每步只问 1 个选择题）

### 第 1 步：目标类型（2 选 1）
> "这个讨论主要想解决什么？"
> - A) 效率问题（太慢/太繁琐）
> - B) 质量问题（易错/易漏）

### 第 2 步：约束条件（是/否）
> "是否有不可改变的限制？"
> - A) 有（如"不改数据库""不换系统"）
> - B) 没有/不确定

### 第 3 步：当前现状（3 选 1）
> "现在的情况更接近？"
> - A) 完全手工（靠人肉）
> - B) 部分自动化（有些功能但不好用）
> - C) 系统有但难用（功能全但体验差）

### 第 4 步：方案偏好（3 选 1）
> "你更倾向哪种方式？"
> - A) 最小改动快速上线（能解决就行）
> - B) 彻底重构一步到位（长期稳定）
> - C) 分阶段迭代（先解决最痛的）

---

## 回答后会发生什么

1. 我召集对应专家讨论
2. 输出方案对比表（核心思路/优点/缺点/边界情况/推荐指数）
3. 列出需要你拍板的关键选择点（<=5 个）
4. 你确认后，写入 `consensus.md` 成为决策

---

## 附录：人机协作流程（2026-03-03 共识，简化版）

> 详细说明：`docs/roundtable/consensus.md`（2026-03-03 AI 人机协作流程决策）
> **简化原则：说人话，AI 追问**

### 新对话启动协议

**每次开启新对话时，按以下顺序操作：**

1. **读取上下文**（必须）
   ```
   @docs/status/current-status.md
   @docs/handover.md
   ```

2. **选择性读取**（按需）
   - 功能开发：`@docs/modules/<模块>/spec.md`
   - 流程相关：`@docs/context/ops-processes.md`
   - Bug 修复：`@docs/quality/system-functional-matrix.md`

### 指令方式：说人话，AI 追问

**你只需要说想做什么，AI 会自动追问缺失的信息。**

**示例：**

```
你：修复供应商 Tab 只读问题

AI：收到。为了准确修复，我需要确认 3 个问题：

1. 是否可以修改后端 API？还是只能用现有接口？
2. 需要支持哪些操作？（编辑/删除/绑定？）
3. 保存失败时需要什么提示？

你：1. 不改后端  2. 编辑和删除  3. 友好错误提示

AI：明白，开始修复...
```

**为什么这样设计：**
- 你不需要记模板
- AI 主动追问缺失信息
- 自然对话完成结构化

### 参考模板（可选）

**如果你想写得更清楚，可以参考这个格式（不是必须）：**

```markdown
[做什么] 一句话
[限制] 不可改什么（可选）
[验收] 怎么算完成（可选，1-2 条）
```

**示例：**
```markdown
[做什么] 修复供应商 Tab 只读问题
[限制] 不改后端 API
[验收] 能编辑、能保存、保存后显示新价格
```

### AI 自审查流程

**AI 生成代码后，必须输出审查报告：**

```markdown
## AI 自审查报告

### 代码质量
- [ ] 接口定义符合现有规范
- [ ] 代码量 < 200 行（如超过已拆分）
- [ ] 无硬编码值（配置已提取）

### 异常处理
- [ ] 网络超时处理
- [ ] 数据校验
- [ ] 错误提示友好

### 测试覆盖
- [ ] 正常路径测试用例
- [ ] 异常路径测试用例
- [ ] 边界条件测试用例
```

### 文档同步机制

**代码变更后 24 小时内：**

1. AI 主动提示需要更新的文档
2. 人类审核变更建议
3. 更新以下文件（如适用）：
   - `docs/status/current-status.md` - API 列表、Bug 列表
   - `docs/modules/<模块>/spec.md` - 功能 spec
   - `docs/context/ops-processes.md` - 流程变更

### Token 优化技巧

| 做法 | 说明 |
|------|------|
| 用 @文件 引用 | 代替文字描述代码结构 |
| 一对话一功能 | 不混谈多个主题 |
| 用 Checklist 审查 | 不让 AI 逐行解释代码 |
| 上下文保持精简 | 过时内容及时归档 |

### 意图边界说明

**`/Bug` vs `/调试`**
- `/Bug`：报告已知问题 → 记录到 status，分配优先级
- `/调试`：定位问题原因 → 找到根因，给出修复建议

**`/开发` vs `/重构`**
- `/开发`：开发新功能 → 追问约束 → 生成代码 → 自审查
- `/重构`：优化现有代码 → 分析问题 → 重构方案 → 执行

### 注意事项

1. **意图词不是必须的**：直接说自然语言也可以
2. **AI 会追问**：如果信息不足，AI 会主动追问
3. **可以打断**：任何时候可以打断当前流程
4. **可以跳过**：如果 AI 追问的问题你不想回答，可以说"跳过"
