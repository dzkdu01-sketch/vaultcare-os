| 确认内容 | 标题/短描述/长描述/分类/标签全部已确认 | 状态 identified → content_ready [状态机 T-P3] | [API §2.2] |
| 执行素材检查 | 已上传图片/视频 | 检查数量/比例/大小/时长，仅返回提示，不自动处理 | [API §2.4] |
| 主图人工确认 | 已选定主图 | 主图确认状态更新 | [API §2.4] |
| 确认素材 | 素材强校验通过 + 主图已确认 | 状态 content_ready → asset_ready [状态机 T-P4] | [API §2.2] |
| 提交为可分销 | G-P5 门禁全部通过 | 状态 asset_ready → distributable [状态机 T-P5] | [API §2.2] |
| 挂起 | 任意非终态 | 状态 → suspended，记录挂起原因 [状态机 T-P6] | [API §2.2] |
| 回退 | 非草稿状态 | 回退到上一状态 [状态机 T-P8~11] | [API §2.2] |

**门禁可见性**：
- 顶部状态进度条始终显示当前卡住门禁 + 缺失项
- 内容区每个字段都要显示“未生成 / 待确认 / 已确认 / 已手动修改”状态
- 素材区把“强拦截问题”和“提示问题”分层展示
- 按钮禁用时，悬停必须显示不可点击原因
- “AI 生成并确认”与“人工直接填写”都视为合法确认来源

**权限差异**：
- 老板 / 运营：可编辑、可确认、可触发 AI、可提交状态流转 [权限 R1-P2, R2-P2]
- 分销商：不可见此页面 [权限 R3-P2]

## 商品模块原型专项说明 `[v3]`

### 1. 为什么页面 1 要支持行内确认

因为批量规范化和存量商品全量刷新后，如果每一条都必须跳详情页，会导致确认成本过高。因此页面 1 必须承担“快速确认台”的职责。

### 2. 为什么页面 2 不自动处理图片

本期业务决策已确认：素材系统只做提示，不自动裁切、不自动压缩、不自动替换。原因是素材内容合规、构图和主图判断仍需人工负责，自动处理容易引入不可控风险。

### 3. 为什么 AI 不单独成为主状态

AI 生成只是辅助动作，不应该让主状态机膨胀。主状态仍只保留业务阶段；AI 的“未生成 / 待确认 / 已确认”只作为字段级状态存在，并服务于 G-P3 / G-P4 门禁。

### 4. 存量商品刷新在原型中的体现

页面 1 的“批量 AI 规范化”不仅服务新导入商品，也服务历史存量商品。确认后的状态先停留在“已确认未同步”，不自动推送 WordPress。
