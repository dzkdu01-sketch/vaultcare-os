## 2) 指标定义

### A. token 消耗（代理指标）

- 单次 `/开发` 启动上下文字数（以首轮上下文块字符量估算）

- 重复段落数量

### B. 交接效率

- 接手到“可开始编码”的分钟数（样本会话估算）
- 首轮必读文档数量

## 5) 结论（v1）

- token 目标（下降 30%）：
  - 直接代理“上下文字数”当前为 -28.8%，接近目标；
  - 结构代理（引用文档数、重复段落）已明显改善。
- 交接时间目标（下降 35%）：
  - v1 估算达到 -38.9%，暂判“达标（低置信度）”。

## 6) 主要有效动作（Top 3）

1. 清理入口文档重复段落，减少重复注入。
2. 固化 SSOT 与“每次开发回填最多 3 处”规则，降低文档扩散。
3. 建立周治理节奏模板，形成持续校准闭环。

## 7) 本周问题与下周动作

- 遗留问题：
  - token 与交接时间仍以估算为主，缺少自动采样脚本支撑。
- 下周动作（最多 3 条）：
