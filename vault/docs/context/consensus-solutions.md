# Vaultcare OS 已达成共识与解决方案

## 核心架构方案：星型解耦架构 (Hub and Spoke)

为了实现"高扩展、低维护、强防封"的目标，系统采取"前端展示与后端管理彻底分离"的架构。

### 宏观架构脑图 (Mindmap)

```text
========================================================================================
                               Vaultcare OS 宏观架构图
========================================================================================

【大脑层】Vaultcare 中台 (Hub) -- (定制开发，独立纯净后台，对FB绝对隐身)
  │
  ├── [1] 全局商品中心 (Master PIM)
  │    ├── 统一 Master SKU 库 (标题, 描述, 主图图床链接)
  │    ├── 供应商映射矩阵 (QR编码/成本/库存 vs VIP编码/成本/库存)
  │    └── 分销商选品权 (各分销商在Hub勾选自己想卖的商品，加入专属"选品清单")
  │
  ├── [2] 智能订单中心 (OMS)
  │    ├── 快捷录单端 (专为"消息互动"设计: 极简输入手机+SKU，自动计算利润，快速出单)
  │    ├── 订单池 (汇总来自极简录单和各WP网站的订单)
  │    └── 智能路由引擎 (根据规则自动决定推单给 VIP 还是 QR，优先 VIP)
  │
  ├── [3] 站点环境调度器 (Site Manager) - 资产绝对隔离
  │    ├── 自营1组: [特货域名1] + [独立白页1] + [像素1] + [WhatsApp 1] + [支付方式1] + (选品清单)
  │    ├── 自营2组: [特货域名2] + [独立白页2] + [像素2] + [WhatsApp 2] + [支付方式2] + (选品清单)
  │    ├── 分销商A: [域名A] + [独立白页A] + [像素A] + [WhatsApp A] + [支付方式A] + (选品清单)
  │    └── 分销商B: [域名B] + ...
  │    (中台统一配置 WhatsApp，自动更新到前端所有商品页)
  │
  └── [4] 财务对账看板 (Dashboard)
       ├── 老板全局视角: 总单量、总利润、QR/VIP欠款追踪
       └── 分销商视角: 只能看到自己选品清单的订单状态(发货/签收/拒收)、利润核算

                                 ↑ (双向加密 API 通信) ↓
========================================================================================
                             网络隔离层 (防火墙 + Cloudflare)
========================================================================================
                                 ↑ (推单回传)           ↓ (库存同步 & 资产配置下发)

【触角层】独立前端集群 (Spokes) -- (高危阵地，全部用轻量 WordPress，随时可弃可重建)
  │
  ├── [隔离环境 1] 自营 WP 站 1 (装载斗篷插件 -> 指向独立白页1)
  ├── [隔离环境 2] 自营 WP 站 2 (装载斗篷插件 -> 指向独立白页2)
  ├── [隔离环境 A] 分销商 A 的 WP 站 (装载斗篷插件 -> 指向独立白页A)
  └── [隔离环境 B] 分销商 B 的 WP 站 ...

  (前端WP站特点：不允许后台人工添加/修改商品，商品全是中台推过来的"镜像"；
   Hub一键下架，所有相关WP站自动同步下架（Draft，页面消失）)
```

### 核心机制解答

1. **商品下架同步：**
   Hub 是所有库存的唯一真相来源。当老板在 Hub 把 SKU 状态改为"下架/无货"时，Hub 会立即通过 API 触发 webhook 到所有绑定了该 SKU 的前端 WordPress 站点，把该商品状态变更为 **Draft（草稿）** —— 页面从前端完全消失，非 Out of Stock。分销商无需操作。

2. **分销商选品自由度：**
   商品数据全部存在 Hub。在 Hub 里，每个分销商有一个"选品清单"。分销商登录 Hub 看板，从全局商品库中勾选自己想卖的商品加入清单。只有这部分商品会被推送到他自己的 WordPress 网站上。

3. **分销商资产管理：**
   防封隔离的核心。Hub 的"站点环境池"模块独立管理每个分销商/自营站的资产包：特货域名(A站)、白页域名(B站)、Pixel ID、WhatsApp 号码。

4. **WhatsApp 动态绑定：**
   中台统一配置，下发前端。在 Hub 给"分销商 A"配置其 WhatsApp 号码。当 Hub 向其 WP 推送商品页面时，会自动把这些页面的 WhatsApp 聊天链接替换成分销商 A 的号码。如果 WhatsApp 被封，只需在 Hub 后台改一次号码，前端所有商品页的链接瞬间自动更新。

## 补充共识（方案评审后确认）

> 以下为 2026-03-02 方案深度评审后新增的已达成共识。

### 5. 供应商路由引擎 — 熔断机制

路由引擎包含"供应商熔断"逻辑。当供应商（如 QR）出现资金链断裂、暂停服务等问题时，老板在中台一键触发熔断，路由引擎自动将所有新订单全切到其他可用供应商（如 VIP）。无需逐单手动调整。

### 6. 站点环境调度器 — 支付方式隔离

站点环境资产包中新增"支付方式"字段。每个分销商/自营组应配置独立的广告支付方式（虚拟信用卡等），避免因共用支付方式导致 FB 关联封禁。

### 7. 斗篷服务 — 共用服务器，独立配置

斗篷服务共用同一台服务器，但每个分销商/自营组使用独立的配置文件（独立白页域名、独立拦截规则）。不需要为每个环境部署独立斗篷实例。

### 8. 快捷录单端 — 分销商自操作

快捷录单端支持 Web 和手机浏览器访问，由分销商自己操作（非老板或审单员）。分销商在 WhatsApp 与客户确认订单后，自行在录单端录入。

### 9. 范围排除（已确认暂不纳入）

以下内容经讨论后确认暂不纳入系统范围：
- 供应商风险仪表盘（VC 库存扣押等）
- 资金回流通道管理（钱庄路径）
- WhatsApp Business API 自动化集成
- 100 对话 → 客户筛选的 UX 流程优化

### 10. 技术选型确认

| 维度 | 选择 |
|------|------|
| 后端 | Python 3.12 + Django 5.x + Django REST Framework |

### 11. 商品编码体系

**自动编码格式：** `vc-{region}-{category}{seq:03d}`

示例：`vc-u-1001`（UAE / Vibrators / 第001号）

| 字段 | 规则 |
|------|------|
| 固定前缀 | `vc` |
| 区域代码 | `u`=UAE, `t`=泰国, `a`=全区域 |
| 品类代码 | 单字符 `1`-`9` 或 `A`（共10个品类） |
| 序号 | 3位数字（001-999），按(区域+品类)独立计数 |

**区域代码语义（重要）：**
- 编码中的区域代码 = 商品**创建时的主要目标市场**（不可变，不代表销售限制）
- `MasterSKU.region` 字段 = **当前允许推送的市场范围**（可更新）
- 分销商可跨区域选品（UAE编码商品可推给泰国分销商）

**旧商品处理：** 保留旧码（如 `QR42`）作为 `master_code` 初始值，存入 `legacy_code`；管理员可按需升级为规范编码。

### 13. MasterSKU 模型新增字段

相比原有模型，新增以下字段：

| 新增字段 | 类型 | 说明 |
|---------|------|------|
| `legacy_code` | CharField | 旧码（QR42等），可搜索 |

原有 `WPProductMapping` 缺乏同步状态追踪，需重建，新增字段：

| 新增字段 | 说明 |

| `wp_sku` | WP SKU字段值（如QR42） |

| `sync_error` | 失败原因文本 |

### 15. WP 同步策略（V1 范围已确认）

- **V1 只同步上下架状态**，不同步价格/描述/图片
- 商品下架 → WP 商品变为 **Draft（页面消失）**，非 Out of Stock
- 同步延迟：**分钟级可接受**，使用异步任务队列
- 商品下架级联：自动删除分销商选品 + 触发 WP 下架

### 16. AI 辅助上线模块

- 使用 **Claude API**（已有 API Key），推荐 claude-3-5-haiku 模型
- AI 只提建议，人工最终确认，禁止 AI 直接创建商品
- 功能：图片 → 自动填写 title_en / title_ar / short_description / description / 品类 / 标签
- 含阿拉伯语质量评估（海湾方言适配）
- 预期效率：5分钟/SKU（原流程 20-30 分钟）

### 17. 分销商商品可见性规则

- 分销商**可以看到成本价**（设计要求，非安全漏洞）
- 显示当前最优可用供应商的成本价（路由引擎同款逻辑）
- API 新增 `availability` 聚合字段：`available` / `low_stock` / `unavailable`

### 18. WordPress CSV 批量导入

- 来源：`backend/pim/商品导出模板.csv`（vaultcare-d.com 主站）
- 目标：**只导入主站**，分销商站点通过选品流程自行推送
- 旧商品保留旧码（不强制重新编码），可后续手动升级
- 导入脚本自动清洗：视频URL从description剥离、分类映射、Tags拆分

### 19. 暂不纳入范围（V1 明确排除）

- 库存数量同步（只同步上下架状态）
- 价格/描述/图片的 WP 内容同步（V2）
- 商品变体（Variant/SKU变体）
- 供应商库存 API 接入
- WhatsApp 推广文案生成、素材包与导出（**V3 纳入**）
