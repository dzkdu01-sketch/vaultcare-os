# 圆桌 v2 开场协议（子项目版）

> 版本：2.0 | 更新：2026-03-04
> 设计思路：**上下文注入 → 发散想法 → 提取想法 → 固化规则**
> 核心升级：专家先推荐、用户确认/增补；标准输出模板；飞轮指标回流

---

## 一、开场：专家推荐与确认

### 1.1 流程

```
用户：/圆桌 [主题]
   ↓
AI：输出「推荐专家名单 + 理由 + 缺失风险」
   ↓
用户：确认 / 增补 / 调整
   ↓
未确认 → 不进入正式讨论
已确认 → 进入「发散想法」阶段
```

### 1.2 推荐名单表单（AI 输出格式）

```markdown
## 专家推荐

**问题类型：** [效率/质量/流程/架构/混合]

**推荐名单：**

| 序号 | 专家角色 | 必要性（一句话） |
|------|----------|------------------|
| 1 | 流程专家 | 负责流程拆解和边界识别 |
| 2 | AI 协作专家 | 负责人机分工和追问策略 |
| 3 | 执行角色（如审核员） | 负责落地视角和约束条件 |

**若缺某角色可能导致：** [简要说明漏角色的风险]

**可增补位：** 你可追加 1–2 位业务/领域专家（如 PM、分销商、商品管理员）

请确认名单：[确认 / 增补：角色名 / 调整：说明]
```

### 1.3 用户确认方式

| 用户说 | 含义 |
|--------|------|
| 确认 | 按推荐名单进入讨论 |
| 增补：PM | 在现有名单上增加 PM |
| 调整：去掉审查专家 | 移除某位，再进入讨论 |
| 跳过 | 使用默认推荐（不推荐，仅在紧急时） |

## 二、标准流程：上下文注入 → 发散 → 提取 → 固化

### 2.1 阶段 0：业务上下文注入（防瞎出主意）

**目标：** 确保所有专家基于真实的业务流程、痛点和约束进行思考，避免脱离实际的空想。

**AI 行为：**
- **主动读取**：自动检索并读取相关文档（如 `docs/process-registry.md`、`docs/status/current-status.md`）
- **强制追问**：若缺失必要上下文，必须向用户确认以下至少一项：
  1. 当前的真实业务流程是怎样的？
  2. 核心痛点/阻碍是什么？
  3. 有什么不可打破的客观约束（如成本、合规、老系统）？
- **输出**：在专家开始发散前，AI 必须先输出一段「**已拉齐的业务上下文**」供用户确认或补充。

### 2.2 阶段 A：发散想法（AI 主动发散）

**目标：** 多视角、无约束地产生观点和方案，**AI 必须承担主力发散角色，不能依赖用户抛出想法**。

**AI 行为强制要求：**
1. **必须基于上下文**：专家发言必须引用或针对「已拉齐的业务上下文」，禁止脱离实际的套话。
2. **主动保底输出**：AI 必须主动提出**至少 3 个截然不同维度的解决方向**（例如：最快落地版、彻底重构版、业务降级版、跨界借鉴版等）。
3. **引入极端/对立视角**：强制分配一位专家扮演「红队（唱反调）」或「极端约束」角色（如：“如果预算/时间减半怎么做？”、“这个方案最容易崩的点在哪？”），打破惯性思维。
4. **禁止过度依赖用户**：在未给出至少 3 个有价值的差异化方案雏形前，**绝对禁止反问用户“您有什么想法？”或“您倾向哪个？”**。必须先由 AI 提供充足的“弹药”，将填空题变为选择题/改错题。
5. **碰撞式输出**：以每位确认专家的第一人称口吻，呈现观点的**碰撞与摩擦**，而非一团和气。允许矛盾观点并存，不急于收敛。

### 2.3 阶段 B：提取想法

**目标：** 从发散内容中提炼可决策的结构化产出。

**AI 必须输出：**

| 产出项 | 要求 |
|--------|------|
| 方案对比表 | ≥3 个方案，含核心思路/优点/缺点/推荐指数 |
| 待确认问题 | ≤5 个，封闭式优先 |
| 建议决议 | 1 句，若可给出 |

**标准模板：**

```markdown
## 方案对比表

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 核心思路 | ... | ... | ... |
| 优点 | ... | ... | ... |
| 缺点 | ... | ... | ... |
| 推荐指数 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

## 待确认问题

1. [封闭式问题，可选 是/否 或 列表选项]
2. ...
3. ...

## 建议决议

[一句话，若可给出]
```

### 2.4 阶段 C：固化规则

**目标：** 将用户确认的决策写入规范，供后续复用。

**触发条件：** 用户明确说「确认」「写入共识」「进入下一阶段」等。

**AI 行为：**
1. 将决策写入 `docs/roundtable/consensus.md`
2. 若涉及意图流程，更新 `docs/roundtable/intents/` 对应文档
3. 记录到 `docs/roundtable/sessions/` 本场纪要
4. 生成变更摘要（写了哪些文件）

## 三、飞轮指标最小集（验证回流）

### 3.1 圆桌质量指标

| 指标 | 计算方式 | 用途 |
|------|----------|------|
| 专家组合有效性 | 会后用户反馈：「本次专家组合有效/无效」 | 优化专家路由规则 |
| 待确认问题数量 | 每场 ≤5 个 | 控制收敛成本 |
| 决策执行率 | 写入 consensus 的决策中，已执行数/总数 | 衡量决策落地 |

### 3.2 人机协作整体指标（与现有飞轮对齐）

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| 追问轮次 | 平均每意图的追问次数 | 发散-提取是否高效 |
| 完成率 | 意图流程完成数 / 启动数 | 固化规则是否被正确执行 |
| 返工率 | 用户说「不对」的次数 / 总轮次 | 提取是否准确 |

### 3.3 数据收集

- **每场圆桌结束：** AI 在 session 纪要末尾追加「本场飞轮备注」
- **每周 /飞轮报告：** 汇总指标，生成优化建议

**本场飞轮备注模板：**

```markdown
## 本场飞轮备注

- 专家组合：流程专家、AI 协作专家、审核员
- 专家组合有效性：[待用户反馈：有效/无效]
- 待确认问题数：3
- 决策数量：2
- 写入文件：consensus.md、04-圆桌.md
```

## 四、与现有规则的关系

| 现有规则 | 本协议 | 关系 |
|----------|--------|------|
| 圆桌不主动退出 | 阶段 C 仅在用户明确指令后执行 | 一致，强化 |
| 专家路由规则（04-圆桌.md） | 1.1 推荐名单 | 升级为「先推荐再确认」 |
| 方案对比表 3 个、待确认 ≤5 | 2.2 提取阶段 | 一致，标准化模板 |
| consensus.md 写入 | 2.3 固化规则 | 一致，明确写入时机 |
