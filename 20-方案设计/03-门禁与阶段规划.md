# Vaultcare 门禁与阶段规划

> 日期：2026-03-08
> 状态：讨论版
> 作用：用于评估项目能否进入下一阶段，以及开发前必须卡住哪些规则
> 上位文档：`01-中台总方案.md`

---

## 1. 为什么要单独写门禁

这次中台如果不先卡门禁，最容易发生 3 个问题：

1. 范围不断膨胀，最后做成泛 ERP。
2. 页面先做出来，规则和状态机后补，最后返工。
3. 业务口径、原型口径、开发口径三套话，最后没人能判断什么算完成。

所以这份文档的目标就是：

**把“能不能往下走”写清楚。**

## 2. 阶段门禁

### Gate 0：立项门

先确认这次到底要解决什么，不解决什么。

必须回答：

- 这次最核心的业务目标是什么
- 一期一定做什么
- 一期坚决不做什么
- 成功怎么衡量

验收标准：

| 验收项 | 标准 | 验证方式 |
|---|---|---|
| 业务目标 | 5 条北极星目标已确认且无歧义 | 老板签字确认 |
| 一期范围 | 做什么 / 不做什么清单已冻结 | 文档版本锁定 |
| 成功指标 | 至少 3 条可量化的成功指标已定义 | 写入方案文档 |
| 上位输入 | 三份业务底稿已通过审查且无未解决冲突 | 审查记录存档 |

未通过时：

- 不允许进入页面设计和研发拆解

对应阶段：P0（范围与口径冻结）入口

### Gate 1：对象门

先确认关键对象和边界，不然系统一定乱。

必须回答：

- 主商品是什么，是否包含供应商关联和货权类型（自营 / 外部）
- 分销商是不是经营单元，主档是否包含关联 WhatsApp 联系账号
- 域名站点和 webhook 是否是独立对象，webhook 是否双向（发布 + 订单回传）
- 订单来源已明确：WordPress webhook 自动接入 + 非网站渠道手动录入，其中手动录入需支持 `manual` 与 `whatsapp` 两类来源标记
- 订单交接记录是否独立
- 供应商 / 货盘 / 自营商品是否拆开，边界是否清晰

验收标准：

| 验收项 | 标准 | 验证方式 |
|---|---|---|
| 商品身份 | 主商品定义已确认，含供应商关联和货权类型字段 | 对象定义文档 |
| 分销商 | 分销商作为经营单元已确认，含 WhatsApp 联系账号字段 | 对象定义文档 |
| 域名站点 | 域名、双向 webhook 作为独立对象已确认 | 对象定义文档 |
| 订单来源 | WordPress 自动接入 + 非网站人工录入路径已定义，且人工录入支持 `manual` / `whatsapp` 来源标记 | 流程定义文档 |
| 供应商 / 货盘 / 自营 | 三者边界已定义，商品主档包含供应商和货权类型 | 对象定义文档 |
| 交接与拒收 | 交接记录和拒收任务作为独立对象已确认 | 对象定义文档 |

未通过时：

- 不允许进入流程设计

对应阶段：P0（范围与口径冻结）出口

### Gate 2：流程门

先确认主流程和异常流，不然页面只是空壳。

必须回答：

- 商品怎么变成 `可分销`
- 分销怎么变成 `已发布`
- 订单怎么从来源接入进入 `待审核`（区分 WordPress 自动接入和非网站人工录入路径，人工录入下可区分 `manual` 与 `whatsapp` 来源标记）
- 审单规则至少定义：按供应商拆单触发条件、按市场和货权判断履约路由（货盘(QR) / 货盘(VIP) / 泰国自营仓库）、阻断异常类型
- 异常和拒收怎么闭环（拒收任务关闭标准不依赖未建设的完整结算系统，使用最小化拒收结算标记；订单关闭前需经过结算确认）
- UAE 和 TH 两个市场的履约模式和货权模式差异已定义

验收标准：

| 验收项 | 标准 | 验证方式 |
|---|---|---|
| 商品状态机 | `草稿 → 入池 → 身份确认 → 内容整理 → 素材确认 → 可分销` 完整定义，含回退路径和挂起 | 状态机图 |
| 分销状态机 | `选品 → 待发布 → 发布中 → 已发布 / 失败` 完整定义，含重试和撤销 | 状态机图 |
| 订单状态机 | `来源接入 → [草稿] → 待审核 → 待交接 → 已交接 → 待结算确认 → 关闭` 完整定义，含异常和拒收分支 | 状态机图 |
| 审单规则 | 拆单触发条件、履约路由规则、阻断异常类型已定义 | 规则文档 |
| 拒收闭环 | 拒收任务从创建到关闭、订单进入 `pending_settlement` 的完整流程已定义 | 流程文档 |
| 多市场 | UAE 和 TH 的履约模式与货权模式差异已定义 | 市场配置文档 |

未通过时：

- 不允许进入原型和研发

对应阶段：P1（流程与规则定义）出口

### Gate 4：原型门

必须回答：

## 3. 三个模块的业务门禁

### 3.1 商品模块门禁

#### G-P1 入池门

- 有来源
- 有最小原始资料

#### G-P2 身份确认门

- 主商品身份已建立
- 疑似重复已人工确认
- 主编码已生成或已归并
- 供应商已关联
- 货权类型已标记（自营 / 外部）

#### G-P3 内容整理门

- 标题、卖点、描述、基础价格、分类已确认
- AI 结果已人工确认

#### G-P4 素材确认门

- 至少 1 张可用主图
- 图片尺寸和质量达标
- 主图已人工确认

#### G-P5 可分销门

- 身份、内容、主图、价格、模板规则都满足
- 供应商已关联且货权类型已标记
- 商品状态变为 `可分销`

---

### 3.2 分销模块门禁

#### G-D1 站点可用门

- 站点已绑定分销商
- 域名已配置
- 发布 webhook 已配置并启用
- 订单回传 webhook 已配置并启用
- 站点状态可用

#### G-D2 商品可分销校验门

- 商品状态为 `可分销`
- 市场模板匹配
- 内容与素材达最低要求

#### G-D3 发布执行门

- 关键动作留痕完整
- 可发起发布、撤销、重试

#### G-D4 发布回执门

- 必须拿到结构化回执
- 失败必须能单站重试

---

### 3.3 订单模块门禁

#### G-O1 草稿门（仅适用于非网站渠道手动录入；WordPress 自动接入的订单跳过此门，直接进入 G-O2）

- 有客户基本线索
- 已知商品或商品集合
- 已知来源渠道（WhatsApp / 互动消息等）

#### G-O2 提交待审核门

- 客人明确确认
- 关键信息完整
- 商品信息明确
- 来源站点 / 渠道明确

#### G-O3 审核通过门

- 信息完整
- 拆单判断完成（依据商品供应商归属，不同供应商的商品需拆单）
- 履约承接方式明确（按市场和货权判断：货盘(QR) / 货盘(VIP) / 泰国自营仓库）
- 无阻断异常

#### G-O4 交接门

- 审核通过
- 交接对象明确（货盘(QR) / 货盘(VIP) / 泰国自营仓库，依据履约路由）
- 已生成交接记录

#### G-O5 结果关闭门

- 已有明确结果（签收 / 拒收 / 其他）
- 拒收订单必须生成拒收任务
- 拒收任务关闭前必须标记拒收结算状态（待结算 / 已标记 / 已关闭）
- 关闭原因明确并留痕

## 5. 当前最值得先拍板的 6 件事（已全部确认）

> 确认日期：2026-03-09
> 确认方式：根据业务底稿、需求说明书、状态机定义、技术选型文档综合推导

### 5.1 商品可分销的最小门槛 — 已确认：条件完整，无遗漏

G-P5 通过条件（共 11 项）：

| 序号 | 条件 | 来源门禁 |
|---|---|---|
| 1 | 主编码已生成 | G-P2 |
| 2 | 供应商已关联 | G-P2 |
| 3 | 货权类型已标记（自营/外部） | G-P2 |
| 4 | 标题已确认 | G-P3 |
| 5 | 短描述已确认 | G-P3 |
| 6 | 长描述已确认 | G-P3 |
| 7 | 主分类已确认 | G-P3 |
| 8 | 价格已确认 | G-P3 |
| 9 | 至少 1 张主图且已人工确认 | G-P4 |
| 10 | 图片尺寸和质量达标 | G-P4 |
| 11 | 无待人工确认的 AI 字段 | G-P3 + G-P4 |

以上 11 项全部满足即可标记 `distributable`。

### 5.2 审单规则最小化定义 — 已确认：规则已明确

**拆单触发条件：**
- 订单内商品涉及 ≥2 个不同供应商时触发拆单
- 拆单时机：`reviewing` 阶段，审核通过前完成
- 拆单结果：原订单标记"已拆单（父）"，按供应商生成 N 个子订单
- 子订单起始状态：`pending_handover`

**履约路由规则：**

| 市场 | 货权类型 | 路由目标 | 编码 |
|---|---|---|---|
| UAE | 外部供应商 | 对应货盘 | `qr_warehouse` 或 `vip_warehouse` |
| UAE | 自营 | QR/VIP 仓库 | `qr_warehouse` 或 `vip_warehouse` |
| TH | 自营 | 泰国自营仓库 | `th_warehouse` |
| TH | 外部供应商 | 泰国自营仓库（代收代发） | `th_warehouse` |

### 5.3 WordPress 订单回传机制 — 已确认：使用 WooCommerce Webhook

| 项目 | 方案 |
|---|---|
| 技术路径 | WooCommerce `order.created` webhook |
| 接收端点 | `/api/v1/order/webhook/receive` |
| 认证方式 | HMAC-SHA256 签名验证（`X-WC-Webhook-Signature`） |
| 幂等处理 | `wp_order_id` + `site_id` 去重 |
| 数据处理 | 从 WooCommerce 标准格式提取所需字段，映射到内部模型 |
| 异常处理 | 关键字段缺失 → 进入 `exception`，不进入正常审核 |
| WooCommerce 版本 | 10.1.4 |

### 5.4 拒收任务关闭标准 — 已确认：最小化结算标记够用

- 拒收任务状态链路：`pending → processing → pending_settlement → closed`
- 关闭前必须完成最小化结算标记（待结算 / 已标记 / 已关闭）
- 关闭时必须填写关闭原因
- 特殊情况：无法联系客户时可从 `processing` 直接关闭，需说明原因
- 拒收任务关闭后，订单进入 `pending_settlement`，不直接关闭订单
- 一期不需要完整结算系统，最小化结算标记足以支撑闭环

## 6. 当前建议

如果要保证"快速、简单、可以继续开发"，最合适的推进方式不是直接写开发任务，而是严格按统一阶段体系（P0 → P9）逐步推进，每个阶段必须通过对应 Gate 才能进入下一阶段：

1. **P0：** 先用 Gate 0 + Gate 1 冻结范围和对象定义（含供应商、货权、订单来源、双向 webhook）
2. **P1：** 再用 Gate 2 定义三模块状态机和审单规则
3. **P2：** 再用 Gate 3 补页面字段字典和权限矩阵
4. **P3：** 再用 Gate 4 验证原型主链路
5. **P4：** 再用 Gate 5 确认技术方案（含 WordPress 对接和双向 webhook）
6. **P5-P7：** 按模块开发，每个模块独立过 Gate 6
7. **P8-P9：** 集成验收和试运行
