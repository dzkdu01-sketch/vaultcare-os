## 1. 目标

本次前端项目骨架的目标不是只创建一个可启动的空项目，而是搭建一个可持续开发的高保真中后台前端底座，用于承接 Vaultcare 一期主链路页面与后续真实接口接入。

已确认约束：

- 骨架层级：高保真骨架
- 技术栈：React 19 + TypeScript + Vite + Tailwind CSS 4 + shadcn/ui
- 数据策略：先用 mock 跑通主要页面交互
- 鉴权范围：包含登录页、权限壳、受保护路由

本次设计同时满足三类目标：

1. 能看：页面具备较完整的中后台观感
2. 能演示：核心页面可通过 mock 数据跑通浏览与跳转
3. 能继续开发：未来接真实 API 时不需要推倒页面与目录结构重来

---

## 2. 方案选择

本次采用“标准中后台骨架，先稳住主干”的方案。

方案特点：

- 做成单应用 SPA
- 优先建立应用壳、业务导航、通用页面模板、状态表达和 mock service 边界
- 在骨架阶段完成足够高的视觉保真度，但不过度追求每个业务页面的完整业务实现

不采用其他方案的原因：

- 原型优先方案虽然页面观感更快成型，但后续接真实接口返工风险高
- 平台优先方案抽象更厚，但对一期骨架来说偏重，容易过度设计

## 3. 总体架构

前端骨架定义为一个单仓单应用的中后台 SPA，由五层组成：

### 3.1 应用层

负责系统运行方式：

- 路由
- Provider
- 登录态
- 受保护路由
- 全局壳状态

### 3.2 布局层

负责统一页面框架：

- 登录布局
- 后台主布局 `AppShell`
- 侧边栏
- 顶部栏
- 页面容器
- 面包屑/标题区

### 3.3 业务页面层

负责承接一期主要模块：

- Dashboard
- 商品
- 分销
- 订单
- 系统设置

### 3.4 数据层

负责隔离 mock 与未来真实接口：

- mock service
- API service 占位
- adapter/service 边界

### 3.5 设计系统层

负责把现有文档里的视觉与状态规则落成代码，尤其是：

- 色彩系统
- 状态 Badge
- 表格
- 卡片
- 按钮
- 页面布局
- 间距与字体规范

## 4. 页面与模块拆分

## 7. 数据流与状态设计

### 7.1 数据流

骨架阶段采用：

```txt
Mock Data
  -> Mock Service
  -> Module Hook
  -> Page
  -> UI Component
```

未来替换成：

```txt
API Client
  -> API Service
  -> Module Hook
  -> Page
  -> UI Component
```

页面层尽量不知道自己使用的是 mock 还是真实接口。

### 7.2 状态管理分类

#### 全局状态

用于：

- 登录用户
- 角色
- 侧边栏状态
- 主题
- 运行环境

#### 服务端状态

用于：

- 列表数据
- 详情数据
- 提交结果

建议交给 query 层管理。

#### 页面本地状态

用于：

- 筛选条件
- 弹窗开关
- 当前 tab
- 当前选中行

原则是不把所有状态都塞进一个全局 store。

### 7.3 类型设计

建议区分三层：

- Domain Type
- DTO
- View Model

通过 mapper 层进行转换，避免把后端返回结构直接硬塞给 UI。

### 7.4 状态与视觉映射集中管理

所有状态文案与颜色映射集中在配置中管理，例如：

- 商品状态映射
- 订单状态映射
- 发布状态映射
- 市场映射
- 角色映射

不要把颜色类名散落在各页面。

## 10. 错误处理设计

错误分三层：

### 10.1 页面级错误

例如列表/详情加载失败，表现为页面内错误卡片与重试按钮。

### 10.2 区块级错误

例如详情页某个卡片加载失败，表现为卡片内部错误占位，不拖垮整页。

### 10.3 动作级错误

例如登录失败、保存失败、发布失败，表现为字段提示、局部提示与 toast 结合。

同时区分：

- 业务问题：门禁不通过、状态不允许推进
- 系统问题：请求失败、超时、未知错误

两者在 UI 表达上不混用。
