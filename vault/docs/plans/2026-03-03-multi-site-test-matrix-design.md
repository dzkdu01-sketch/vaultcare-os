# 多站点（同一账号）功能矩阵与测试体系设计稿

> 日期：2026-03-03  
> 背景：北京分销商朋友同一账号下同时运营 UAE 站与 TH 站；需要以“站点”为第一维度隔离选品/定价/导出/同步/语言与可售范围。  
> 目的：统一文档与沟通规则，让需求→评估→测试→验收→沉淀进入可重复的飞轮。

---

## 1. 文档体系（方案 A）

### 1.1 母表（唯一真相）
- 文件：`docs/quality/system-functional-matrix.md`
- 定位：系统功能全景账本。任何功能/缺失/缺陷都必须在母表有且仅有一条记录。
- 结构：三层
  - L1 模块
  - L2 子模块
  - L3 流程功能（可测试/可验收的最小单位）

### 1.2 角色测试剧本（可执行步骤）
- 目录：`docs/test-scenarios/`
- 核心剧本：
  - `docs/test-scenarios/distributor-multi-site.md`
  - `docs/test-scenarios/admin-ops.md`
- 原则：每条步骤必须回链母表条目 ID，确保“测试结果可追踪、可复测”。

### 1.3 圆桌纪要（灵感池）
- 文件：`docs/product-order-brainstorming.md`
- 定位：保留为纪要/背景，不承担“状态真相”；状态以母表为准。

---

## 2. 母表字段规范（每个 L3 必填）

- **ID**：建议 `模块缩写-层级编号`（例：`PIM-1.1.3`、`OMS-2.1.1`、`DIST-3.2.2`）
- **适用站点**：`UAE` / `TH` / `ALL`
- **角色**：`系统管理员` / `商品管理员` / `分销商` / `销售&ERP`（可多选）
- **频率**：`高频` / `低频`
- **状态（统一枚举）**
  - 🔴 Bug：已实现但行为错误/不生效
  - 🟡 缺失-待评估开发：目前缺失，需要评估合理性/依赖/优先级
  - 🟣 已实现-待验证：从代码/状态文档看已做，但缺少可复现的验收证据
  - 🟢 已验证OK：已按剧本验证通过，必须附证据
  - ⚪ 暂缓/不做：明确不在范围或已被替代（写原因）
- **证据（强制规则）**
  - 只有填写证据，才能把状态标为 🟢
  - 证据格式建议：`测试人/日期/环境/剧本步骤范围/截图或接口`

---

## 3. “同一账号多站点”的功能边界

### 3.1 必须站点隔离（默认必须）
- 币种与价格展示/编辑（UAE=AED；TH=THB）
- 导出/同步到站点（导出数据包、同步队列、站点商品映射）
- 语言内容（标题/描述/素材包语言版本）
- 可售范围/合规（某些 SKU 仅允许特定站点）

### 3.2 默认共享（可升级为站点隔离）
- MasterSKU 主档基础信息
- 供应商/库存聚合（如未来需要按站点仓库维度再拆分）

---

## 4. 圆桌 2（四轮）固定议程与产出

参与者：ERP专家×1、PM×1、分销商×2、系统管理员×2

- 第1轮（现状&高频链路）：拉通多站点高频链路（选品→导出→同步→下单）
- 第2轮（缺失补齐）：补齐运营常见需求（导入/批量标签/QR独有/站点隔离定价等）
- 第3轮（PM+ERP初评）：对 🟡 缺失项给出合理性、依赖、建议优先级与替代方案
- 第4轮（验收口径）：把每个 🔴 Bug 的验收标准固化进剧本与母表

---

## 5. 沟通口令扩展（写入飞轮手册）

- `对齐目标`：输出目标/非目标/成功标准/角色/站点维度一页纸，用户只做是/否修正
- `评估文档`：基于母表规范裁决现有文档是否可作为基线，并给出修订清单
- `圆桌2`：按四轮议程输出结论，并自动回写母表与剧本
- `生成测试清单-分销商`：生成分销商多站点高频剧本
- `生成测试清单-管理员`：生成管理员/运营剧本（导入导出、批量、字典维护）
