# 圆桌纪要：人机协作门禁与飞轮进化升级

> 日期：2026-03-04  
> 主题：降低用户沟通负担，避免过早进入 UI/开发，强化开发前闭环

---

## 一、问题与目标

### 1.1 现存问题

- 在业务流程未闭环前过早讨论 UI，导致返工。
- 在功能口径未清晰时提前进入开发，导致 Bug 增加。
- 用户被迫回答技术细节，协作负担高、节奏慢。

### 1.2 目标

- 让用户只做业务决策，不回答技术实现。
- 开发前形成最小闭环（目标/边界/验收）。
- 通过飞轮持续优化提问质量与门禁效果。

---

## 二、核心决策

| # | 决策项 | 选择 | 说明 |
|---|--------|------|------|
| 1 | 提问语言 | 业务优先 | AI 默认只问业务问题，不要求用户回答技术实现细节 |
| 2 | 信息收集方式 | AI 预填 80% + 用户确认 | AI 先给默认草案，用户仅确认/修订关键点 |
| 3 | 追问负担控制 | 每轮最多 3 个阻断问题 | 非阻断信息后置，避免拉长前置沟通 |
| 4 | 开发前门禁 | 关键缺口可拒绝进入开发 | 缺目标/边界/验收时，AI 必须拦截并给补齐清单 |
| 5 | 阶段约束 | 先闭环后体验 | 业务流程未闭环前，不进入 UI 细节讨论 |

---

## 三、开发前最小闭环清单

1. 业务目标（要提升什么结果）
2. 范围边界（明确不做什么）
3. 主流程与异常流程（至少 1 主线 + 2 异常）
4. 关键业务字段（最小必填集合）
5. 验收标准（建议 3-5 条可验证结果）

---

## 四、飞轮进化补充指标

| 指标 | 用途 |
|------|------|
| 业务问题占比 | 检查是否坚持业务优先提问 |
| 单轮追问数 | 控制沟通负担（目标每轮 ≤ 3） |
| 预填采用率 | 衡量“AI 预填 80%”落地效果 |
| 门禁拦截次数 | 评估门禁执行强度 |
| 拦截后一次通过率 | 评估门禁质量（是否拦得准） |

---

## 五、执行节奏

- 每次 `/开发` 前：先执行“最小闭环清单”
- 每周 5 分钟：复盘门禁拦截 Top 3 场景
- 每周回写：将通过验证的优化回写到 `INTENTS-GUIDE.md` 与 Checklist
