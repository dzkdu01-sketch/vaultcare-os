# AI 协作检查清单（2026-03-03 共识）

> 每次与 AI 协作开发时，按此清单执行。
> 来源：`docs/roundtable/consensus.md`（2026-03-03 AI 人机协作流程决策）

---

## 协作默认模式（2026-03-04 更新）

**默认采用低负担模式：AI 预填，你只确认。**

- [ ] AI 先给出业务草案（目标/边界/流程/验收），用户不从零填写
- [ ] AI 每轮最多追问 3 个“阻断问题”
- [ ] AI 对用户提问使用业务语言（不要求技术实现回答）
- [ ] 若关键前置缺失，AI 可拒绝直接进入 `/开发`
- [ ] 自动直推不覆盖门禁：门禁未齐时，必须先补齐再开发

---

## 开发硬门禁（v1.3 热修）

**进入 `/开发` 前必须全部满足：**

- [ ] 已确认“本轮唯一业务目标”（1 句话）
- [ ] 已确认“本轮不做项”（至少 1 条）
- [ ] 已确认“验收标准”（3-5 条，至少 1 条页面行为）

**阻断规则：**

- [ ] 上述任一缺失，AI 必须拒绝进入开发并给出补齐清单（最多 3 个阻断问题）
- [ ] 用户说“继续/直接做”但门禁未齐时，AI 仍先补门禁
- [ ] 若已进入开发后发现口径不清，先执行飞轮纠偏，再恢复开发

---

## 阶段 1：对话启动

**每次开启新对话时执行：**

- [ ] 让 AI 读取 `@docs/status/current-status.md`
- [ ] 让 AI 读取 `@docs/handover.md`
- [ ] 按需读取相关文件（模块 spec、流程文档等）
- [ ] 确认 AI 理解当前项目状态（让它复述关键点）

---

## 阶段 2：指令下达

**说人话即可，AI 会先预填再追问：**

- [ ] 告诉 AI"想做什么"（一句话）
- [ ] 优先确认 AI 预填草案（同意/修改 1-2 条）
- [ ] 回答 AI 的业务追问（目标、边界、验收）
- [ ] 确认 AI 理解正确后，再让它开始写代码

**示例对话：**
```
你：修复供应商 Tab 只读问题

AI：收到。我先给默认草案（目标/边界/验收），你只需确认：
1. 目标优先“稳定”还是“效率”？
2. 本轮不做哪些项？
3. 哪条业务结果必须达成？

你：效率优先，不改流程角色，草稿态+图片+品类三条都必须达成

AI：明白，开始修复...
```

**参考格式（可选，不是必须）：**
```markdown
[做什么] 一句话
[限制] 不可改什么（可选）
[验收] 怎么算完成（可选）
```

---

## 阶段 3：代码生成

**AI 生成代码时的要求：**

- [ ] AI 先输出流程设计/伪代码（人类确认后再写代码）
- [ ] 代码量控制在 200 行内（超过必须拆分）
- [ ] 小步迭代，每步验证

---

## 阶段 4：AI 自审查

**AI 生成代码后，必须输出审查报告：**

### 代码质量
- [ ] 接口定义符合现有规范
- [ ] 代码量 < 200 行（如超过已拆分）
- [ ] 无硬编码值（配置已提取）

### 异常处理
- [ ] 网络超时处理
- [ ] 数据校验
- [ ] 错误提示友好

### 测试覆盖
- [ ] 正常路径测试用例
- [ ] 异常路径测试用例
- [ ] 边界条件测试用例

---

## 阶段 5：文档同步

**代码变更后 24 小时内：**

- [ ] AI 主动提示需要更新的文档
- [ ] 人类审核变更建议
- [ ] 更新以下文件（如适用）：
  - [ ] `docs/status/current-status.md` - API 列表、Bug 列表
  - [ ] `docs/modules/<模块>/spec.md` - 功能 spec
  - [ ] `docs/context/ops-processes.md` - 流程变更

---

## Token 优化技巧

| 做法 | 说明 |
|------|------|
| 用 @文件 引用 | 代替文字描述代码结构 |
| 一对话一功能 | 不混谈多个主题 |
| 用 Checklist 审查 | 不让 AI 逐行解释代码 |
| 上下文保持精简 | 过时内容及时归档 |

---

## Bug 预防清单

**开发完成后自查：**

- [ ] AI 是否追问了缺失信息（如约束、验收标准）
- [ ] AI 是否先给了默认草案，减少用户从零输入
- [ ] AI 追问是否控制在“每轮最多 3 个阻断问题”
- [ ] AI 是否理解了现有代码结构（读取了相关文件）
- [ ] 生成的代码是否有 AI 自审查报告
- [ ] 是否有 3 个核心测试用例
- [ ] 代码是否控制在 200 行内（超过已拆分）
- [ ] 上下文文档是否已更新（如有变更）
