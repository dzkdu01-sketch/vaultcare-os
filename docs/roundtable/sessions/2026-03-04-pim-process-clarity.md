## 三、流程依赖关系简图

```
[商品来源]
  ├── WP CSV 导入 (F3) ──────────────► MasterSKU
  ├── AI 辅助新增 (F1) ─────────────► MasterSKU (草稿→上架)
  └── 手动新增 (F2) ────────────────► MasterSKU

[商品维护]
  编辑(F4) / 批量操作(F5) / 导入导出(F6) / 供应商绑定(F10)
       │
       ▼
  MasterSKU 变更 ──► [下架时] 级联(F7) ──► 清除选品 + WP draft

[分销侧]
  分销商选品(F8) ──► DistributorSelection ──► WPProductMapping(pending)
       │
       ▼
  推送任务(F9) ──► WooCommerce API ──► WP 商品 publish

[可售状态]
  SupplierSKU ──► availability 聚合(F11) ──► 选品/录单可见
```

## 八、圆桌补充评审（商品管理模块适配性，2026-03-04）

> 主题：设计中的商品管理模块是否满足当前业务与流程需求  
> 专家组：流程架构、PIM 产品、前后端实现、运营执行、测试验收、分销商代表、UI/UX 设计专家（用户增补）

### 8.1 结论

采用 **方案 B（流程闭环优先）**：先补齐流程闭环能力与验收证据链，再进行体验重构。

### 8.2 用户确认决策（5 项）

| # | 问题 | 用户确认 |
|---|------|----------|
| 1 | 本轮目标优先级 | 先达成流程闭环，再做体验重构（是） |
| 2 | `PIM-F13` 发布门禁 | 同意采用「草稿 → 待审核 → 可发布」三态 |
| 3 | `PIM-F18` 站点级控制 | 同意全局状态与站点发布状态分离 |
| 4 | `PIM-F17` 导入治理 | V1 先做「批次审计 + 手动修复」 |
| 5 | 验收口径 | 同意以 `system-functional-matrix + DoD` 双证据为准，冲突以复测更新母表 |

### 8.3 落地影响

- 流程策略：`F13/F16/F17` 继续保持优先推进，`F17` 的 V1 目标收敛为“审计优先”。
- 架构策略：多站点能力按“全局商品状态 + 站点发布状态”双层模型设计。
- 验收策略：所有“已实现/待验证”条目必须补齐 matrix 与 DoD 的双证据后才可标绿。

### 8.4 子议题 B 续议（`PIM-F18` 业务口径，2026-03-04）

为避免讨论过度技术化，本轮改为业务决策提问并完成确认：

| # | 业务问题 | 用户确认 |
|---|----------|----------|

| B2 | 是否允许同一商品在不同站点独立经营（A站上架、B站不上架）？ | 允许 |
