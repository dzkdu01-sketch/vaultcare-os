## 1. 一句话定位

Vaultcare 当前不应先建设完整 ERP，而应先建设一套面向多市场复制的**轻量业务中台**，用来把：

`商品建档工作台 -> 分销站点协同 -> 订单运营工作台`

这条一期主链路稳定固化下来，并为未来 ERP / WMS / 财务系统对接预留接口边界。

---

## 2. 为什么不是先做完整 ERP

当前最关键的问题不是“缺一个大系统”，而是：

1. 商品资料分散，难以稳定变成可卖商品。
2. 分销站点依赖零散配置，域名、webhook、发布状态和失败重试不统一。
3. 订单进入人工处理后，草稿、待审核、交接、异常处理依然依赖人脑和表格拼接。
4. 老板的高价值时间被低价值协同工作吞噬。

所以现在最合理的路线不是重型 ERP，而是先把中段高频、重复、可标准化的动作收口。

---

## 3. 一期北极星目标

### 3.1 业务目标

1. 让商品从“资料碎片”进入“可分销商品”的时间明显下降。
2. 让分销协同从“人工发站点”升级到“可配置、可回执、可重试”。
3. 让网站订单进入统一运营链路，形成 `草稿 -> 待审核 -> 待交接 -> 结果关闭` 的闭环。
4. 让多市场复制依赖模板，而不是依赖口头约定。
5. 让老板减少重复协同，把时间回收到商品和投放判断。

### 3.2 系统目标

1. 建立统一的商品身份锚点和商品建档工作台。
2. 建立分销商、域名站点、webhook 和发布关系的统一模型。
3. 建立带草稿状态的订单运营工作台。
4. 建立严格门禁，避免"边开发边改口径"。
5. 预留 ERP / WMS / 财务系统对接边界，但一期不提前做重。

### 3.3 一期可量化成功指标

以下指标用于衡量一期上线后的实际业务改善效果。基线值基于当前业务底稿中的痛点描述进行估算，待一期开发启动前通过实际测量校准。

| 指标 | 衡量方式 | 基线（当前） | 目标（一期上线后） |
|------|----------|-------------|-------------------|
| 商品从资料入池到可分销的平均耗时 | 系统记录：商品创建时间 → 通过可分销门禁时间 | 估算 2–3 天（含人工整理图片、文案、对齐多源资料） | 降低 60%（目标 1 天内） |
| 单个商品资料整理的人工操作步骤数 | 人工统计：整理一个商品所需的跨系统操作次数（打开表格、复制粘贴、下载图片、上传站点等） | 估算 15–20 步（涉及多个表格、素材文件夹、WordPress 后台） | 降低 50%（通过统一工作台 + AI 辅助收口） |
| 订单从网站成交到进入待审核状态的平均时长 | 系统记录：WordPress 下单时间 → 订单进入统一订单池待审核时间 | 估算 2–4 小时（依赖人工发现新订单并手动搬运） | 降低 80%（通过 webhook 自动接入，目标 30 分钟内） |
| 订单信息在不同系统间的重复录入次数 | 人工统计：单笔订单从接入到交接完成，信息被手动复制粘贴的次数（含搬运到分销表格、导出给货盘等） | 估算 3–5 次/单 | 降低 70%（通过统一订单池和结构化交接，目标 ≤1 次） |
| 拒收任务从接收到传递给分销商的平均处理周期 | 系统记录：拒收结果进入系统时间 → 任务分配给对应分销商时间 | 估算 1–2 天（需人工通过订单号反查客户、定位分销商和联系方式） | 降低 50%（通过订单关联分销商 + WhatsApp 联系账号自动定位） |
| 商品发布到分销站点的失败感知时长 | 系统记录：发布请求时间 → 失败状态被运营感知的时间 | 估算 1–2 天（依赖人工逐站点检查，失败常被遗漏） | 降低 90%（通过发布回执 + 失败重试机制，目标实时感知） |
| 老板每周花在商品资料整理和订单协同上的时间占比 | 人工记录：老板每周工作时间中用于资料整理、订单搬运、重复协同的小时数占比 | 估算 40%–50%（大量时间被低价值重复劳动占用） | 降低至 20% 以下（释放时间回到广告素材与投放策略判断） |

说明：

- 以上基线值均为基于业务底稿痛点描述的合理估算，建议在 P0 阶段通过 1–2 周实际跟踪记录进行校准。
- 目标值以百分比改善表示，不设绝对数字，以适应不同阶段的业务规模变化。
- 每条指标均可通过系统埋点数据或人工定期统计进行验证，具体验证方式在 UAT 阶段确认。

## 4. 一期范围

### 4.1 一期一定要做

#### A. 商品模块

- 商品主档
- 主编码与别名体系
- 供应商关联（商品绑定供应商，用于后续拆单和履约路由判断）
- 货权类型标记（自营 / 外部供应商，用于区分履约承接方式）
- 商品建档工作台
- AI 辅助翻译 / 文案规范化 / 单个优化 / 批量优化
- 图片尺寸和质量处理
- 素材治理
- 可分销门禁

#### B. 分销模块

- 分销商主档（含关联 WhatsApp 联系账号，用于拒收任务时快速定位联系渠道）
- 域名站点管理
- webhook 配置管理（双向：发布推送 + 订单回传）
- 选品关系
- 商品到站点的发布关系
- 发布状态回执
- 失败重试
- 分销关键动作留痕

#### C. 订单模块

- WordPress 订单自动接入（通过站点订单回传 webhook 进入统一订单池）
- 非网站渠道手动录入入口（WhatsApp / 互动消息等渠道的订单，由运营手动创建草稿）
- 统一订单池
- 订单草稿
- 草稿提交待审核
- 审单（含最小化审单规则：按供应商拆单、按市场判断履约路由）
- 拆单支持
- 交接记录（含履约路由：货盘(QR) / 货盘(VIP) / 泰国自营仓库）
- 异常订单
- 拒收任务
- 拒收结算标记（最小化，不是完整对账系统，仅标记拒收订单的结算状态以支持任务关闭）
- 订单核心动作留痕

### 4.2 一期明确不做

- 完整财务总账
- 完整采购管理
- 完整仓储系统
- 广告投放系统
- 全渠道 CRM
- 重 BI / 数据仓库
- 复杂分佣和渠道结算
- 完整账号资产管理系统（但分销商主档中保留最小化的 WhatsApp 联系账号字段，用于拒收任务快速定位）

### 4.3 一期预留接口但不落重

- ERP 主数据同步
- WMS 库存和履约状态回写
- 财务对账 / 回款记录同步
- 经营看板事件流

## 5. 三模块关系

### 5.1 模块定位

- 商品模块：主模块，把资料稳定变成可分销商品
- 分销模块：放大器，把商品发到具体域名站点
- 订单模块：反馈器，把站点成交后的订单拉回统一运营闭环

### 5.2 最小闭环

```mermaid
flowchart LR
    productWorkbench[商品建档工作台]
    sellableProduct[可分销商品]
    distributorModule[分销站点模块]
    sitePublish[域名与Webhook发布]
    orderDraft[订单草稿]
    orderReview[待审核订单]
    orderOps[订单运营处理]
    resultLoop[结果回流]

    productWorkbench --> sellableProduct
    sellableProduct --> distributorModule
    distributorModule --> sitePublish
    sitePublish --> orderDraft
    orderDraft --> orderReview
    orderReview --> orderOps
    orderOps --> resultLoop
    resultLoop --> productWorkbench
```

## 6. 多市场原则

多市场复制必须按模板能力设计，不把 UAE / TH 规则直接写死在流程里。

一期至少要先定义：

1. 市场模板
2. 履约模式
3. 货权模式
4. 审核门槛模板
5. 字段规则模板

### 一期落地方式

一期不单独建设市场模板管理页面，但在以下位置内嵌最小化的市场配置能力：

- 商品主档：目标市场字段（UAE / TH）
- 分销商主档：所属市场字段
- 订单审核：按市场判断履约路由（货盘(QR) / 货盘(VIP) / 泰国自营仓库）
- 订单交接：按市场路由到对应承接侧

完整的市场模板管理页面和规则引擎作为二期能力预留。

## 7. 阶段规划

以下为统一阶段体系，同时覆盖业务模块推进和开发生命周期（详细门禁见《门禁与阶段规划》）。

### P0：范围与口径冻结

产出：

- 方向确认
- 一期范围冻结
- 核心对象定义（含供应商、货权类型、订单来源）
- 门禁规则定义

对应门禁：Gate 0（立项门）+ Gate 1（对象门）

### P1：流程与规则定义

产出：

- 商品 / 分销 / 订单三模块状态机
- 审单规则最小化定义（拆单触发条件、履约路由规则）
- 拒收任务闭环流程（含关闭标准）
- UAE / TH 市场差异最小化定义

对应门禁：Gate 2（流程门）

### P2：数据与权限定义

产出：

- 8 页面字段字典
- 权限矩阵（老板 / 运营审单 / 分销商）
- 留痕范围确认
- 订单草稿与待审核的责任边界

对应门禁：Gate 3（数据门）

### P3：原型验证

产出：

- 8 页面原型
- 商品 / 分销 / 订单三条主链路走通验证
- 门禁可见性验证（用户能看到为什么被卡住）

对应门禁：Gate 4（原型门）

### P4：技术拆解

产出：

- 技术方案（复用 / 新建判断）
- API 草图
- WordPress 订单回传技术方案
- 双向 webhook 技术方案
- 回滚方案

对应门禁：Gate 5（技术门）

### P5：商品模块开发

目标：

- 统一来源入池
- 统一商品身份（含供应商关联和货权类型）
- AI 辅助
- 素材治理
- 统一可分销门槛

对应门禁：Gate 6-商品

### P6：分销模块开发

目标：

- 管理分销商（含 WhatsApp 联系账号）
- 管理域名站点与双向 webhook
- 打通选品、发布、回执、失败重试

对应门禁：Gate 6-分销

### P7：订单模块开发

目标：

- WordPress 订单自动接入
- 非网站渠道手动录入
- 草稿、待审核、审单、交接
- 异常与拒收任务（含拒收结算标记）

对应门禁：Gate 6-订单

### P8：集成与 UAT

目标：

- 三模块联调
- 业务角色实操验收
- 主链路端到端走通

对应门禁：Gate 7（UAT 门）

### P9：试运行

目标：

- 灰度上线
- 监控部署
- 培训完成
- 回滚预案就绪

对应门禁：Gate 8（试运行门）

## 8. 对外系统边界

### ERP 未来更适合承接

- 采购
- 库存台账
- 财务核算
- 标准主数据同步

### WMS 未来更适合承接

- 仓内库存
- 出入库
- 发货状态
- 履约节点回传

### 财务系统未来更适合承接

- 对账
- 回款
- 费用归集
- 利润核算

### 这个中台自身应保留的核心

- 商品主数据和建档工作台
- 分销站点协同
- 订单运营编排
- 规则与门禁
- 经营事件回流

---

## 9. 当前建议结论

对 Vaultcare 来说，最稳妥的路线不是“先做大系统”，而是：

1. 用轻量业务中台先固化真实经营闭环。
2. 用严格门禁约束口径、范围和开发推进。
3. 让一期同时覆盖 `商品 + 分销 + 订单`，但主轴先放在商品建档工作台。
4. 让文档、设计、开发都围绕同一条最小闭环推进。
