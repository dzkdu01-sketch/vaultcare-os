# Vaultcare 一期技术选型与回滚方案

> 日期：2026-03-09
> 状态：需求确认版（v2）
> 作用：定义一期技术选型决策、约束条件、可替代方案和回滚策略，作为 Gate 5（技术门）的核心交付物
> 上位文档：`06-API草图.md`

---

## 1. 选型总览

| 序号 | 决策项 | 选型结论 | 约束等级 |
|---|---|---|---|
| 1 | WordPress 订单回传认证 | HMAC-SHA256 签名验证 | 硬约束（依赖 WooCommerce） |
| 2 | AI 服务 | 多供应商路由 + AiService 抽象层（页面 S1 配置） | 软约束（可换供应商） |
| 3 | 图片存储 | OSS + CDN | 软约束（可换云厂商） |
| 4 | 批量操作策略 | 异步任务队列（Redis + BullMQ） | 软约束（可换队列实现） |
| 5 | 发布 webhook 数据格式 | WooCommerce Product API 兼容格式 | 硬约束（依赖 WooCommerce） |
| 6 | 订单回传 webhook 数据格式 | WooCommerce Order Webhook 标准格式 | 硬约束（依赖 WooCommerce） |
| 7 | API 鉴权 | JWT（内部用户）+ HMAC（webhook） | 软约束（可换鉴权方案） |
| 8 | API 版本管理 | URL 路径前缀 `/api/v1/` | 无约束 |
| 9 | 供应商管理 | 商品模块下独立 CRUD API | 无约束 |

> 硬约束：切换需要改动外部系统（WordPress 站点插件），成本较高
> 软约束：切换只需改动平台内部实现，成本可控
> 无约束：纯内部决策，随时可调整

## 2. 逐项选型说明

### 2.1 WordPress 订单回传认证方式

**选型：** HMAC-SHA256 签名验证

**工作原理：**
- 每个站点配置 webhook 时生成一个 secret key
- WordPress 发送请求时，用 secret 对 payload 做 HMAC-SHA256 签名，放入请求头 `X-WC-Webhook-Signature`
- 平台侧接收端点验签，签名不匹配则拒绝

**约束条件：**
- 依赖 WooCommerce 插件（非纯 WordPress），站点必须安装并启用 WooCommerce
- secret key 必须安全存储（数据库加密或环境变量），泄露则需要轮换
- 不防重放攻击（如需防重放，需额外加 timestamp + nonce 校验）

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| API Key（Header 传递） | 实现简单 | 安全性低，key 泄露风险高，无法验证 payload 完整性 | 内部测试环境 |
| IP 白名单 | 无需改动 WordPress 侧 | 共享主机 IP 不固定，多站点维护成本高 | 站点部署在固定 IP 的独立服务器 |
| OAuth 2.0 | 安全性最高，支持 token 刷新 | 实现复杂，WordPress 侧需要额外插件，一期过重 | 二期如果接入非 WordPress 站点 |

### 2.2 AI 服务方案

**选型：** 多供应商路由 + `AiService` 抽象层，供应商和模型由页面 S1 动态配置

**架构：**
```
页面 2 AI 按钮 → 后端 AiService 抽象层 → 查询任务路由配置（页面 S1）
  → 按任务类型路由到对应供应商 + 模型
  → 翻译/文案优化/规范化 → Kimi / DeepSeek / OpenAI 等（文本类）
  → 图片识别 OCR → 智谱 GLM 等（视觉类）
  → 图片处理增强 → Nano Branna 等（图片处理类）
```

**调用方式：**
- 平台后端通过 `AiService` 接口调用，接口定义：翻译、优化、规范化、OCR、图片增强
- `AiService` 读取系统配置（页面 S1 的任务路由表），按任务类型路由到对应供应商
- 所有供应商统一走 OpenAI 兼容接口（大多数国内供应商均支持），自定义供应商走独立适配器
- 批量任务通过异步队列逐条调用，结果写入数据库
- 供应商连接信息（API 密钥、主机、路径）加密存储，运营角色不可见密钥

**约束条件：**
- 依赖外部 API 可用性和延迟（各供应商响应时间不同，一般 2-10 秒/条）
- API 调用有成本，不同供应商定价不同，批量操作需控制成本
- 阿拉伯语和泰语翻译质量因供应商而异，需要实测验证
- 部分地区可能存在 API 访问限制，需确认服务器部署地区的网络可达性
- 供应商配置变更（页面 S1）立即生效，正在执行的批量任务不受影响（使用任务创建时的快照配置）

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 写死单一供应商（原方案） | 实现简单 | 无法按任务类型选择最优供应商，切换需改代码 | 仅使用一个供应商时 |
| 自部署开源模型（Llama 3 等） | 无 API 调用成本，数据不出境 | 需要 GPU 服务器，运维成本高，多语言质量不如商用模型 | 数据合规要求严格、调用量极大时 |
| 专业翻译 API（DeepL / Google Translate） | 翻译质量稳定，价格低 | 只能翻译，不能做文案优化和规范化 | 可作为翻译环节的补充供应商 |

**建议：** 一期通过页面 S1 配置供应商和模型，`AiService` 抽象层确保切换供应商只需改配置、不需改代码。建议初始配置：文本类用 Kimi 或 DeepSeek（性价比高），OCR 用智谱 GLM（视觉能力强），图片增强用 Nano Branna 或同类服务。

### 2.3 图片存储方案

**选型：** 云对象存储（OSS）+ CDN

**架构：**
```
前端 → 获取预签名上传 URL → 直传 OSS → OSS 回调通知后端 → 后端记录元数据
用户访问 → CDN 边缘节点 → 回源 OSS
```

**约束条件：**
- 需要云厂商账号和 CDN 配置（阿里云 OSS + CDN 或 AWS S3 + CloudFront）
- CDN 需要覆盖 UAE 和 TH 两个地区的边缘节点
- 图片处理（裁剪、压缩）建议用 OSS 自带的图片处理服务，避免自建
- 存储成本随图片量线性增长，需设置生命周期策略（如原始图保留 90 天，处理后图长期保留）

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 本地存储 + Nginx | 零外部依赖，部署简单 | 无 CDN 加速，UAE/TH 访问慢，扩容困难，无高可用 | 开发和测试环境 |
| 自建 MinIO | 兼容 S3 API，可私有部署 | 需要自行运维，无 CDN，需额外配置 | 数据不能上公有云时 |
| WordPress 媒体库 | 无需额外存储系统 | 图片分散在各站点，无法统一管理，不适合中台定位 | 不推荐 |

### 2.4 批量操作并发上限和超时策略

**选型：** 异步任务队列

**参数：**

| 参数 | 值 | 说明 |
|---|---|---|
| 单次批量请求上限 | 50 条 | 超过 50 条需分批提交 |
| 单用户并发批量任务数 | 3 个 | 防止单用户占满队列 |
| 单条 AI 处理超时 | 60 秒 | 超时标记为失败，不阻塞后续 |
| 单条图片处理超时 | 30 秒 | |
| 整批任务超时 | 30 分钟 | 超时未完成的条目标记为失败 |
| 前端轮询间隔 | 5 秒 | 通过 task_id 查询进度 |
| 失败重试次数 | 2 次 | 单条失败自动重试，重试间隔 10 秒 |

**约束条件：**
- 需要 Redis 作为消息队列后端（BullMQ 依赖 Redis）
- 任务状态需持久化到数据库（Redis 重启不丢失已完成的结果）
- 批量任务的进度需要实时可查（前端轮询或 WebSocket 推送）

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 数据库轮询队列 | 无需 Redis，实现简单 | 轮询有延迟，高并发下数据库压力大 | 批量操作频率低、团队不熟悉 Redis 时 |
| 同步处理 + 长超时 | 实现最简单 | 请求阻塞，用户体验差，超时风险高 | 批量数量极少（<5 条）时 |
| RabbitMQ / Kafka | 功能更强，支持复杂路由 | 运维复杂度高，一期用不到高级特性 | 消息量极大或需要复杂消费模式时 |

### 2.5 发布 webhook 推送数据格式

**选型：** WooCommerce Product API 兼容格式

**最小字段集：**

```json
{
  "action": "create | update | delete",
  "product_code": "string (平台侧商品主编码)",
  "name": "string",
  "description": "string",
  "short_description": "string",
  "regular_price": "string",
  "sale_price": "string",
  "tags": [
    { "name": "string" }
  ],
  "images": [
    { "src": "string (CDN URL)", "position": 0 }
  ],
  "categories": [
    { "name": "string" }
  ],
  "featured": false,
  "in_stock": true,
  "weight": "string",
  "status": "publish | draft",
  "meta_data": [
    { "key": "vaultcare_product_code", "value": "string" },
    { "key": "vaultcare_supplier", "value": "string" }
  ]
}
```

> `[v2]` 新增 `sale_price`（实际售价）、`tags`（标签）、`featured`（是否推荐）、`in_stock`（库存状态）、`weight`（重量），对齐 `05-页面字段字典.md` §11.1 商品导入导出字段映射。

**约束条件：**
- 格式绑定 WooCommerce 生态，非 WooCommerce 站点需要开发适配器
- 站点侧需要安装一个轻量 WordPress 插件来接收推送并调用 WooCommerce REST API
- 图片 URL 必须是公网可访问的 CDN 地址（WordPress 会下载到本地媒体库）
- `meta_data` 中的 `vaultcare_product_code` 用于建立平台商品与站点商品的映射关系

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 自定义格式 + 站点适配器 | 格式自由，不受 WooCommerce 限制 | 每种站点系统都需要开发适配器，一期工作量大 | 二期接入非 WordPress 站点时 |
| 直接调用 WooCommerce REST API | 不需要站点侧插件 | 平台需要存储每个站点的 WooCommerce API 凭证，安全管理复杂 | 站点数量少且信任度高时 |

### 2.6 订单回传 webhook 数据格式

**选型：** WooCommerce Order Webhook 标准格式（平台侧提取所需字段）

**工作流程：**
1. WooCommerce 站点配置 `order.created` webhook，目标 URL 指向平台 `/api/v1/order/webhook/receive`
2. 新订单创建时，WooCommerce 自动推送完整订单 JSON
3. 平台侧接收端点从 WooCommerce 标准格式中提取所需字段，映射到内部订单模型
4. `06-API草图.md` 中定义的请求体结构作为内部存储格式，不是 webhook 接收格式

> `[v2]` 内部存储格式已更新：新增客户邮箱、WhatsApp、结构化地址（姓名+地址+城市+国家）、支付方式、三段式金额（商品小计+配送费=订单总额）、订单归因（流量来源/设备类型/会话页面数）、客户IP。详见 `05-页面字段字典.md` §11.3 和 `06-API草图.md` §4.1。

**约束条件：**
- WooCommerce webhook payload 较大（完整订单 JSON 约 5-15KB），平台侧只提取部分字段
- 需要处理 WooCommerce 的 webhook 重试机制（失败后会重试，需做幂等处理，用 `wp_order_id` + `site_id` 去重）
- 不同 WooCommerce 版本的 webhook payload 格式可能有细微差异，需做容错处理

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 自定义精简格式（站点插件推送） | payload 小，只推送需要的字段 | 需要开发站点侧插件，增加站点维护成本 | 对 payload 大小敏感或需要自定义字段时 |
| 定时拉取（轮询 WooCommerce REST API） | 不需要 webhook 配置，实现简单 | 有延迟（取决于轮询间隔），API 调用量大 | webhook 不可靠或站点防火墙限制出站请求时 |
| 混合模式（webhook + 定时兜底） | 实时性好，且有兜底保障 | 实现复杂度高 | 二期如果发现 webhook 丢单时 |

### 2.7 API 鉴权方式

**选型：** JWT（内部用户）+ HMAC-SHA256（webhook 端点）

**JWT 设计：**

| 参数 | 值 |
|---|---|
| 签名算法 | HS256（对称）或 RS256（非对称） |
| access_token 有效期 | 24 小时 |
| refresh_token 有效期 | 7 天 |
| token payload | `{ sub, role, distributor_id?, exp, iat }` |

**鉴权矩阵：**

| 端点类型 | 鉴权方式 | 说明 |
|---|---|---|
| 内部业务 API | JWT Bearer Token | Header: `Authorization: Bearer {token}` |
| 发布回执回调 | HMAC-SHA256 | Header: `X-WC-Webhook-Signature` |
| 订单回传回调 | HMAC-SHA256 | Header: `X-WC-Webhook-Signature` |
| 登录接口 | 无鉴权 | `/api/v1/auth/login` |
| 健康检查 | 无鉴权 | `/api/v1/health` |

**约束条件：**
- JWT secret key 必须安全存储，泄露需要立即轮换并使所有已发 token 失效
- 分销商角色的 JWT 中必须携带 `distributor_id`，后端所有查询自动附加过滤条件（数据隔离）
- refresh_token 需要存储在数据库中（支持主动吊销）
- 一期不做细粒度权限控制（如接口级别），只做角色级别（老板/运营/分销商）

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| Session + Cookie | 实现简单，服务端可控 | 不适合前后端分离架构，跨域问题多 | 传统服务端渲染应用 |
| API Key（全局） | 实现最简单 | 无法区分用户身份和角色，安全性低 | 纯内部工具、无角色区分时 |
| OAuth 2.0 + OIDC | 标准化程度最高，支持第三方登录 | 实现复杂，一期不需要第三方登录 | 二期如果需要分销商独立登录系统时 |

### 2.8 API 版本管理策略

**选型：** URL 路径前缀 `/api/v1/`

**规则：**
- 一期所有 API 统一使用 `/api/v1/` 前缀
- 非破坏性变更（新增字段、新增端点）直接在 v1 上迭代
- 破坏性变更（删除字段、修改字段类型、修改业务语义）引入 `/api/v2/`，v1 保持兼容运行至少 3 个月
- 一期不需要额外投入

**约束条件：**
- 无显著约束

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| Header 版本（`Accept: application/vnd.vaultcare.v1+json`） | URL 更干净 | 调试不直观，客户端实现复杂 | API 消费者多且版本切换频繁时 |
| 查询参数（`?version=1`） | 实现简单 | 不够规范，缓存策略复杂 | 不推荐 |
| 不做版本管理 | 零成本 | 破坏性变更无法平滑过渡 | 不推荐 |

### 2.9 供应商管理方式

**选型：** 商品模块下独立 CRUD API，页面 2 内联创建

**已在 `06-API草图.md` 中落实：**
- `GET/POST /api/v1/product/suppliers` — 列表和新增
- `GET/PUT /api/v1/product/suppliers/{id}` — 详情和更新
- 最小数据模型 9 个字段
- 维护入口：商品详情页（页面 2）来源与身份区，支持选择已有或内联新建

**约束条件：**
- 供应商数据与商品模块耦合，如果后续供应商管理复杂度增加（如供应商评级、合同管理），需要独立为供应商模块
- 一期没有独立的供应商管理页面，供应商数据只能通过商品详情页维护

**可替代方案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 仅作为商品关联字段（无独立 API） | 最简单 | 无法跨商品复用供应商信息，订单模块引用困难 | 供应商数量极少（<5 个）时 |
| 独立供应商模块 + 管理页面 | 管理能力完整 | 一期工作量增加，超出轻量中台定位 | 二期如果供应商管理需求增加时 |
| 对接外部供应商系统 | 数据统一 | 依赖外部系统，增加集成复杂度 | 已有独立供应商管理系统时 |

## 3. 回滚方案

### 3.1 回滚原则

- 每个模块独立回滚，不影响其他模块
- 数据库迁移脚本必须提供对应的 rollback 脚本
- 回滚不删除已产生的业务数据，只切换流量到旧流程
- 回滚决策由老板确认，运营执行

### 3.2 部署策略

推荐蓝绿部署：

```
生产环境（蓝）← 当前流量
预发布环境（绿）← 新版本部署 + 验证
切换：DNS / 负载均衡切换流量到绿
回滚：切换流量回蓝（秒级）
```

一期规模小，也可以用更简单的方式：单机部署 + 数据库备份 + 应用版本回退。

### 3.3 按模块回滚策略

#### 商品模块

| 场景 | 回滚动作 | 数据处理 |
|---|---|---|
| 商品工作台功能异常 | 关闭功能开关，回退到旧的表格管理方式 | 已录入的商品数据保留在数据库，可通过 API 或数据库导出 |
| AI 辅助功能异常 | 关闭 AI 功能开关，人工手动处理 | AI 已生成的结果保留，人工确认流程不受影响 |
| 状态机逻辑异常 | 修复后重新部署，不回滚状态机 | 异常状态的商品由运营手动修正 |

#### 分销模块

| 场景 | 回滚动作 | 数据处理 |
|---|---|---|
| 发布 webhook 异常 | 关闭自动发布，回退到手动登录 WordPress 后台发布 | 已发布的商品不受影响（站点侧数据独立） |
| 站点管理异常 | 站点配置数据保留，手动维护 webhook 配置 | |
| 发布回执接收异常 | 关闭回执接收端点，发布状态由运营手动更新 | |

#### 订单模块

| 场景 | 回滚动作 | 数据处理 |
|---|---|---|
| webhook 订单接入异常 | 关闭接收端点，回退到人工从 WordPress 后台搬运订单 | 已接入的订单继续在新系统中处理 |
| 审核/交接流程异常 | 修复后重新部署 | 流程中的订单由运营手动推进或等待修复 |
| 拆单逻辑异常 | 关闭自动拆单建议，改为纯人工判断 | 已拆的子订单保留 |

#### AI 配置模块 `[补]`

| 场景 | 回滚动作 | 数据处理 |
|---|---|---|
| 多供应商路由异常 | 在页面 S1 将所有任务路由切回单一可用供应商 | 已生成的 AI 结果保留 |
| 某供应商 API 不可用 | 在页面 S1 将受影响任务路由切换到备用供应商 | 正在执行的批量任务中失败条目标记为失败，可重试 |
| AI 配置页面本身异常 | 关闭页面 S1 功能开关，使用数据库中最后一次有效配置 | 供应商配置数据保留在数据库 |
| OCR / 图片增强功能异常 | 关闭对应 AI 按钮（功能开关），回退到纯人工处理 | 已处理的结果保留 |

### 3.4 数据库回滚

| 操作 | 说明 |
|---|---|
| 上线前 | 完整数据库备份 |
| 迁移脚本 | 每个 migration 必须有对应的 rollback（down） |
| 回滚执行 | 按逆序执行 rollback 脚本 |
| 数据保护 | 回滚脚本只回退表结构，不删除业务数据（通过 `ALTER TABLE` 而非 `DROP TABLE`） |
