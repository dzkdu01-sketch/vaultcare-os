# Vaultcare 一期状态机定义

> 日期：2026-03-09
> 状态：需求确认版（v2）
> 作用：定义商品 / 分销 / 订单三模块的状态流转规则，作为 Gate 2（流程门）的核心交付物
> 上位文档：`01-中台总方案.md`、`03-门禁与阶段规划.md`

---

## 0. 评估摘要

从现有页面设计和门禁定义中推导状态机时，发现以下缺口：

1. **商品模块缺少"挂起"和"下架"状态** — 页面 2 有"挂起"按钮，但门禁 G-P1~G-P5 中没有定义挂起状态和回退路径
2. **分销模块缺少"撤销后"状态** — 页面 4 有"撤销"按钮，但撤销后商品回到什么状态未定义
3. **订单模块"驳回"后去向不明** — 页面 7 有"驳回补充"按钮，但驳回后订单回到草稿还是待审核未定义
4. **拆单后子订单的状态机未定义** — 订单模块支持拆单，但拆出的子订单是否共享同一状态机未说明
5. **~~异常状态可在多个节点触发，但异常的解除条件未定义~~** — 已补充：3.7 节定义了异常任务状态机（pending → processing → closed），T-O14 定义了订单异常解除路径
6. **三模块之间的状态依赖未显式定义** — 例如商品从"可分销"变为"挂起"时，已发布的分销关系怎么处理

以下为完整状态机定义，已补入上述缺口的建议方案。

## 1. 商品模块状态机

### 1.1 状态定义

| 状态 | 编码 | 含义 | 对应门禁 |
|---|---|---|---|
| 待入池 | `draft` | 商品刚创建，尚未提交入池 | — |
| 已入池 | `pooled` | 已有来源和最小原始资料 | G-P1 通过 |
| 身份已确认 | `identified` | 主编码已生成，供应商已关联，货权类型已标记 | G-P2 通过 |
| 内容已整理 | `content_ready` | 标题、短描述、长描述、价格、分类已确认，AI 结果如有已人工确认 | G-P3 通过 |
| 素材已确认 | `asset_ready` | 图片/视频校验通过，主图已人工确认 | G-P4 通过 |
| 可分销 | `distributable` | 所有门禁通过，可被分销模块选品 | G-P5 通过 |
| 已挂起 | `suspended` | 人工主动挂起，暂停流转但不丢失已有数据 | — |

### 1.2 状态转换

| 序号 | 从 | 到 | 触发动作 | 条件 | 备注 |
|---|---|---|---|---|---|
| T-P1 | `draft` | `pooled` | 提交入池 | G-P1 通过 | |
| T-P2 | `pooled` | `identified` | 确认身份 | G-P2 通过 | |
| T-P3 | `identified` | `content_ready` | 确认内容 | G-P3 通过 | |
| T-P4 | `content_ready` | `asset_ready` | 确认素材 | G-P4 通过 | |
| T-P5 | `asset_ready` | `distributable` | 提交为可分销 | G-P5 通过 | |
| T-P6 | `pooled` / `identified` / `content_ready` / `asset_ready` / `distributable` | `suspended` | 挂起 | 人工操作 | 从 `distributable` 挂起时触发分销侧级联处理（见 1.4） |
| T-P7 | `suspended` | 挂起前状态 | 恢复 | 人工操作 | 恢复到挂起前的状态 |
| T-P8 | `identified` | `pooled` | 回退身份 | 人工操作 | 发现身份判断有误时回退 |
| T-P9 | `content_ready` | `identified` | 回退内容 | 人工操作 | 内容需要重新整理 |
| T-P10 | `asset_ready` | `content_ready` | 回退素材 | 人工操作 | 素材需要重新处理 |
| T-P11 | `distributable` | `asset_ready` | 撤销可分销 | 人工操作 | 触发分销侧级联处理（见 1.4） |

### 1.3 状态流转图（文本表示）

```
draft → pooled → identified → content_ready → asset_ready → distributable
                      ↑              ↑               ↑              ↓
                      └──回退──┘     └──回退──┘      └──撤销可分销──┘
                                                           ↕
                                                       suspended
                                              (pooled~distributable 均可挂起)
```

### 1.3A 内容与素材确认的子状态规则 `[v3]`

为避免主状态机过重，AI 生成与人工确认不单独提升为主状态，而作为 `identified` → `content_ready` 与 `content_ready` → `asset_ready` 之间的门禁判断依据。

#### 内容门禁（G-P3）通过条件

以下字段必须达到“已确认”状态：

- 标题
- 短描述
- 长描述
- 主分类
- 标签（如适用）
- 价格

字段确认来源允许两种：

1. 人工直接填写
2. AI 生成后由人工确认

只要上述字段全部确认，`G-P3` 即可通过。

#### 素材门禁（G-P4）通过条件

以下检查项必须全部通过：

- 图片数量达标
- 图片比例达标
- 图片大小达标
- 视频时长/格式达标（如有视频）
- 主图已人工确认

图片检查仅做提示，不自动处理；因此素材门禁通过仍依赖人工补图/换图/确认。

#### AI 字段状态（用于页面交互，不进入主状态机）

| 字段状态 | 含义 |
|---|---|
| 未生成 | 还未触发 AI |
| 待确认 | AI 已生成但未人工确认 |
| 已确认 | 人工确认采用该结果 |
| 已手动修改 | 人工在 AI 基础上修改后保存 |

#### 与主状态机的关系

- `identified` → `content_ready`：要求字段级确认完成，而不是只要求“内容存在”
- `content_ready` → `asset_ready`：要求素材校验项全部达标且主图已人工确认
- 运营确认 AI 结果后，等同于完成对应门禁要求，不再额外增加状态流转步骤



| 商品状态变更 | 分销侧影响 |
|---|---|
| `distributable` → `suspended` | 该商品所有"待发布"的分销关系自动取消；已发布的保持不变但标记警告 |
| 非 `distributable` 状态 → `suspended` | 分销侧无影响（该商品尚未进入分销流程） |
| `distributable` → `asset_ready`（撤销可分销） | 该商品所有"待发布"的分销关系自动取消；已发布的保持不变但标记警告 |
| `suspended` → 恢复 | 不自动恢复分销关系，需人工重新选品 |

## 2. 分销模块状态机

分销模块有两层状态：站点状态和发布关系状态。

### 2.1 站点状态

| 状态 | 编码 | 含义 |
|---|---|---|
| 配置中 | `configuring` | 站点已创建，webhook 尚未全部配置完成 |
| 可用 | `active` | G-D1 通过，站点可用于选品和发布 |
| 已停用 | `disabled` | 人工停用，不可用于新的选品和发布 |

转换：

| 从 | 到 | 触发 | 条件 |
|---|---|---|---|
| `configuring` | `active` | 完成配置 | G-D1 通过（域名、双向 webhook 均已配置并启用） |
| `active` | `disabled` | 停用 | 人工操作 |
| `disabled` | `active` | 启用 | G-D1 仍然满足 |
| `active` | `configuring` | webhook 失效 | 系统检测到 webhook 不可用 |

### 2.2 发布关系状态

| 状态 | 编码 | 含义 | 对应门禁 |
|---|---|---|---|
| 已选品 | `selected` | 商品已被选入该分销商的站点 | — |
| 待发布 | `pending_publish` | 已通过可分销校验，等待发布 | G-D2 通过 |
| 发布中 | `publishing` | 正在向站点推送 | G-D3 通过 |
| 已发布 | `published` | 收到站点成功回执 | G-D4 通过 |
| 发布失败 | `publish_failed` | 收到站点失败回执或超时 | — |
| 已撤销 | `revoked` | 人工撤销发布关系 | — |

### 2.3 发布关系状态转换

| 序号 | 从 | 到 | 触发 | 条件 |
|---|---|---|---|---|
| T-D1 | `selected` | `pending_publish` | 校验通过 | G-D2 通过（商品可分销、市场匹配、内容素材达标） |
| T-D2 | `pending_publish` | `publishing` | 发起发布 | G-D3 通过（留痕完整） |
| T-D3 | `publishing` | `published` | 收到成功回执 | G-D4 通过（结构化回执） |
| T-D4 | `publishing` | `publish_failed` | 收到失败回执或超时 | — |
| T-D5 | `publish_failed` | `publishing` | 重试 | 单站重试 |
| T-D6 | `published` | `revoked` | 撤销 | 人工操作 |
| T-D7 | `pending_publish` | `revoked` | 撤销 | 人工操作 |
| T-D8 | `revoked` | `selected` | 重新选品 | 人工操作，需重新走 G-D2 校验 |
| T-D9 | `selected` | `revoked` | 取消选品 | 人工操作 |
| T-D10 | `publish_failed` | `revoked` | 放弃 | 人工操作 |

### 2.4 状态流转图（文本表示）

```
selected → pending_publish → publishing → published
    ↑           ↓                ↓            ↓
    └─重新选品─ revoked ←───── 撤销 ←──── 撤销
                  ↑
           publish_failed
              ↓     ↑
              └─重试─┘
```

## 3. 订单模块状态机

### 3.1 主订单状态

| 状态 | 编码 | 含义 | 对应门禁 |
|---|---|---|---|
| 草稿 | `draft` | 非网站渠道手动录入，尚未提交 | G-O1 通过 |
| 待审核 | `pending_review` | 等待运营审单 | G-O2 通过（手动录入）或 WordPress 自动接入 |
| 审核中 | `reviewing` | 运营正在审核 | — |
| 待交接 | `pending_handover` | 审核通过，等待交接给履约侧 | G-O3 通过 |
| 已交接 | `handed_over` | 已交接给货盘(QR) / 货盘(VIP) / 泰国自营仓库 | G-O4 通过 |
| 已签收 | `delivered` | 用户签收 | — |
| 待结算确认 | `pending_settlement` | 已达到可结算节点，等待运营完成结算确认 | — |
| 已拒收 | `rejected` | 用户拒收，自动生成拒收任务 | — |
| 已关闭 | `closed` | 订单最终关闭 | G-O5 通过 |
| 异常 | `exception` | 标记为异常，需人工处理 | — |
| 已驳回 | `returned` | 审核驳回，退回补充信息 | — |

### 3.2 订单来源与初始状态

| 来源 | 编码 | 初始状态 | 说明 |
|---|---|---|---|
| WordPress webhook 自动接入 | `website` | `pending_review` | 跳过草稿，直接进入待审核 |
| WhatsApp 消息渠道 | `whatsapp` | `draft` | `[v2]` 从 WhatsApp 沟通渠道获取的订单，人工录入草稿 |
| 其他手动录入 | `manual` | `draft` | 需人工创建草稿，确认后提交待审核 |

### 3.3 主订单状态转换

| 序号 | 从 | 到 | 触发 | 条件 |
|---|---|---|---|---|
| T-O1 | — | `draft` | 手动录入 | G-O1 通过 |
| T-O2 | — | `pending_review` | WordPress 自动接入 | webhook 数据校验通过 |
| T-O3 | `draft` | `pending_review` | 提交待审核 | G-O2 通过 |
| T-O4 | `pending_review` | `reviewing` | 开始审核 | 运营领取 |
| T-O5 | `reviewing` | `pending_handover` | 审核通过 | G-O3 通过（信息完整、拆单完成、履约路由明确） |
| T-O6 | `reviewing` | `returned` | 驳回 | 信息不完整或需补充 |
| T-O7 | `returned` | `pending_review` | 补充后重新提交 | 补充信息后重新进入待审核 |
| T-O8 | `pending_handover` | `handed_over` | 交接 | G-O4 通过（交接对象明确、交接记录已生成） |
| T-O9 | `handed_over` | `delivered` | 签收 | 履约侧反馈签收 |
| T-O10 | `handed_over` | `rejected` | 拒收 | 履约侧反馈拒收，自动生成拒收任务 |
| T-O11 | `delivered` | `pending_settlement` | 达到可结算节点 | 签收完成，等待运营结算确认 |
| T-O12 | `rejected` | `pending_settlement` | 拒收任务关闭 | 拒收任务已关闭且拒收结算已标记 |
| T-O13 | `pending_settlement` | `closed` | 结算确认 | G-O5 通过 |
| T-O14 | 任意（除 `closed`） | `exception` | 标记异常 | 人工操作 |
| T-O15 | `exception` | 异常前状态 | 解除异常 | 人工确认异常已处理 |

### 3.4 状态流转图（文本表示）

```
WordPress ──→ pending_review ──→ reviewing ──→ pending_handover ──→ handed_over
                   ↑                 ↓                                  ↓       ↓
手动录入 → draft ──┘            returned                          delivered  rejected
                                    ↓                                ↓         ↓
                              补充后重新提交                 pending_settlement ──┘
                              → pending_review                          ↓
                                                                     closed

                        任意状态 ←→ exception（可标记/可解除）
```

### 3.5 拆单规则

当一个订单包含来自不同供应商的商品时，审核阶段需要拆单：

| 项目 | 规则 |
|---|---|
| 触发条件 | 订单内商品涉及 2 个及以上不同供应商 |
| 拆单时机 | `reviewing` 阶段，审核通过前完成 |
| 拆单结果 | 原订单标记为"已拆单（父）"，生成 N 个子订单 |
| 子订单状态 | 子订单从 `pending_handover` 开始，各自独立流转 |
| 父订单状态 | 父订单进入 `pending_handover` 但不可直接交接，需等所有子订单达到可结算节点后进入 `pending_settlement`，再由运营结算确认关闭 |
| 列表展示 | 父订单在订单列表中保留显示并标记为"已拆单（父）"；子订单独立显示并展示父订单关联标记 |

### 3.6 拒收任务状态

| 状态 | 编码 | 含义 |
|---|---|---|
| 待处理 | `pending` | 拒收任务刚创建 |
| 处理中 | `processing` | 运营或分销商正在跟进 |
| 待结算 | `pending_settlement` | 已达到关闭前条件，等待完成最小化结算标记并进入待关闭确认节点 |
| 已关闭 | `closed` | 任务关闭 |

> 说明：拒收任务的 `pending_settlement` 仍属于任务状态；页面与权限文档中的“拒收结算标记”是独立字段，用于记录最小化结算处理结果，不等同于任务主状态。

转换：

| 从 | 到 | 触发 | 条件 |
|---|---|---|---|
| `pending` | `processing` | 标记处理中 | 人工操作 |
| `processing` | `pending_settlement` | 联系完成 | 人工确认已联系客户 |
| `pending_settlement` | `closed` | 关闭任务 | 已完成最小化结算标记并填写关闭原因 |
| `processing` | `closed` | 直接关闭 | 特殊情况（如无法联系客户），需填写关闭原因并说明原因 |

### 3.7 异常任务状态

异常任务与拒收任务独立，用于处理订单流转中标记为异常的情况（信息缺失、商品异常、客户异常等）。

| 状态 | 编码 | 含义 |
|---|---|---|
| 待处理 | `pending` | 异常任务刚创建 |
| 处理中 | `processing` | 运营正在跟进处理 |
| 已关闭 | `closed` | 异常已解决，任务关闭 |

转换：

| 从 | 到 | 触发 | 条件 |
|---|---|---|---|
| `pending` | `processing` | 标记处理中 | 人工操作 |
| `processing` | `closed` | 关闭任务 | 异常已解决，关闭原因已填写 |
| `pending` | `closed` | 直接关闭 | 误标异常等特殊情况，需填写关闭原因 |

> 注意：异常任务关闭后，关联订单可通过 T-O15 解除异常状态，恢复到异常前状态继续流转。

#### 3.7.1 阻断异常类型

阻断异常用于订单审核阶段（`reviewing`），当存在未解决的阻断异常时，审核不可通过（G-O3 不满足）。

| 枚举值 | 中文显示名 | 触发阶段 | 说明 |
|---|---|---|---|
| `customer_info_incomplete` | 客户信息不完整 | 审核中 | 客户地址、电话等必填信息缺失或格式异常 |
| `product_abnormal` | 商品信息异常 | 审核中 | 订单关联商品已下架、已挂起（suspended）或库存不足 |
| `price_mismatch` | 价格异常 | 审核中 | 订单金额与商品基础价格不匹配，或金额为零/负数 |
| `duplicate_order` | 重复订单 | 审核中 | 系统检测或人工判断为疑似重复下单（同客户+同商品+近时间窗口） |
| `channel_disabled` | 站点/渠道异常 | 审核中 | 订单来源站点已停用（disabled）或来源分销商已停用 |
| `blocking_other` | 其他阻断异常 | 审核中 | 人工标记的其他阻断原因，需在备注中说明 |

> 使用场景：运营审单在 `reviewing` 阶段执行审核通过（T-O5）前，系统自动校验或人工标记阻断异常。所有阻断异常清除后方可通过 G-O3。

#### 3.7.2 异常任务类型

异常任务类型用于订单生命周期中任意阶段（除 `closed`）标记异常（T-O13），创建异常任务时必须选择异常类型。

| 枚举值 | 中文显示名 | 触发阶段 | 说明 |
|---|---|---|---|
| `customer_unreachable` | 客户联系不上 | 审核中 / 待交接 / 已交接 | 多次联系客户未果，无法确认订单信息 |
| `address_change` | 地址变更 | 审核中 / 待交接 / 已交接 | 客户要求变更收货地址，需人工处理 |
| `product_out_of_stock` | 商品缺货 | 审核中 / 待交接 | 订单商品在履约侧确认缺货，无法正常发货 |
| `logistics_abnormal` | 物流异常 | 已交接 | 货盘或仓库反馈物流异常（丢件、延误、派送失败等） |
| `customer_cancel` | 客户取消 | 审核中 / 待交接 / 已交接 | 客户主动要求取消订单 |
| `payment_abnormal` | 支付异常 | 审核中 | COD 订单金额争议、货到付款条件异常等 |
| `exception_other` | 其他异常 | 任意（除 closed） | 不属于以上分类的异常情况，需在备注中说明 |

> 使用场景：运营在订单流转的任意阶段发现异常时，通过 T-O14 标记异常并创建异常任务。异常任务关闭后通过 T-O15 解除订单异常状态。

## 4. 跨模块状态依赖总览

| 上游事件 | 下游影响 |
|---|---|
| 商品 `distributable` → `suspended` | 分销侧：`selected` / `pending_publish` 自动变为 `revoked` |
| 商品 `distributable` → `asset_ready` | 分销侧：同上 |
| 站点 `active` → `disabled` | 分销侧：该站点下所有 `pending_publish` 自动取消 |
| 站点 `active` → `configuring`（webhook 失效） | 分销侧：该站点下所有 `publishing` 标记为 `publish_failed` |
| 分销关系 `published` | 订单侧：该站点可通过订单回传 webhook 接收订单 |
| 订单 `rejected` | 拒收任务自动创建，状态为 `pending` |
| 订单 `exception` | 异常任务自动创建，状态为 `pending` |
| 异常任务 `closed` | 订单可通过 T-O15 解除异常，恢复到异常前状态 |
| 拒收任务 `closed` + 结算已标记 | 订单进入 `pending_settlement` |
| 全部子订单达到可结算节点 | 父订单进入 `pending_settlement` |
